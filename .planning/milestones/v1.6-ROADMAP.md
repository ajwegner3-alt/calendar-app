# Milestone v1.6: Day-of-Disruption Tools

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 31-33
**Total Plans:** 10

## Overview

Give owners two new operational levers for day-of disruption — partial-day unavailability via inverse date overrides and day-level pushback with smart cascade — while hardening the 200/day Gmail SMTP cap into a true refuse-send guard so pushback's higher email volume can't silently exceed it.

**Dependency note:** Phase 31 (hard cap guard) ships first because both AVAIL-06's auto-cancel batch and PUSH-08's pre-flight quota check depend on the guard being live. EMAIL-23 (AVAIL pre-flight UI) is delivered inside Phase 32 and EMAIL-22 (PUSH pre-flight UI) is delivered inside Phase 33; both depend on the Phase 31 guard layer being present. If Phases 32 or 33 were to ship before Phase 31, the auto-cancel and pushback paths would need the existing v1.1 carve-out as a temporary fallback — avoided by the 31 → 32 → 33 execution order.

**Goal:** Owners can mark specific time windows (or the whole day) as unavailable on a date override; the slot engine computes available slots as weekly-hours MINUS unavailable windows; existing bookings inside a new unavailable window are warned about and auto-cancelled on commit.

### Phase 33: Day-Level Pushback Cascade

**Goal:** Owners can push back all bookings from a chosen anchor point forward on a given day by a specified delay, with smart cascade (gap absorption), a pre-commit preview, an optional reason field, and all affected bookings processed through the existing reschedule lifecycle.

**Depends on:** Phase 31 (batch email pre-flight quota check requires EMAIL-21 guard to be live), Phase 32 (no hard dep but natural last in v1.6 — both earlier phases preserve v1.5 invariants that pushback also relies on)

**Requirements:** PUSH-01, PUSH-02, PUSH-03, PUSH-04, PUSH-05, PUSH-06, PUSH-07, PUSH-08, PUSH-09, PUSH-10, PUSH-11, PUSH-12, EMAIL-22

- [x] 33-02-PLAN.md — Pure cascade module + previewPushbackAction + EMAIL-22 quota gate (Wave 2, had human-verify checkpoint)
- [x] 33-03-PLAN.md — rescheduleBooking() skipOwnerEmail+actor extensions + commitPushbackAction with abort-on-diverge race safety (Wave 3)
- [x] 33-04-PLAN.md — Post-commit summary state + per-row retryPushbackEmailAction (Wave 4, had human-verify checkpoint)

**Completed:** 2026-05-06 — Andrew live-verified all 8 scenarios; PUSH-10 gap closed by orchestrator commit `2aa9177`; verifier re-passed

**Details:**

Success Criteria (all met):
1. Owner opens `/app/bookings`, triggers "Pushback," sees a dialog defaulting to today's date listing confirmed bookings chronologically; owner can change the date. ✓
2. Owner picks an anchor booking, enters a delay (positive integer, minutes or hours), optionally types a reason; dialog recalculates new times in real time. ✓
3. Confirmation preview accurately shows: moved (old → new), absorbed (stays), past-EOD bookings; total email count + remaining quota surfaced; if batch would exceed quota, commit disabled. ✓
4. On confirm, every moved booking rides existing reschedule lifecycle: `booking_events` audit row, .ics METHOD:REQUEST SEQUENCE+1, calendar invite update, "sorry for the inconvenience" copy + reason text in email, cancel link. ✓
5. Concurrent booker cancellation/reschedule during pushback commit aborts with ABORT-on-diverge (no partial commits, no duplicate emails); v1.4 EXCLUDE GIST + v1.1 capacity index bind on new times. ✓
6. Owner sees post-commit summary listing per-row status badges (Sent/Failed/Conflict/Stale/Skipped); email failures surface with per-row Retry button without rolling back successful sends. ✓

Key decisions:
- ABORT-on-diverge (NOT union as in Phase 32) — cascade math is order-dependent; any ID set difference aborts entire batch
- `computeCascadePreview()` pure module — zero I/O, zero Supabase, fully unit-testable
- 1440 sentinel for no-rules days (`getEndOfDayMinute` returns 1440) — no false PAST_EOD on fully-blocked days
- Per-booking slot step uses each booking's own `duration_minutes` (not account-wide) — correct boundary snapping
- Quota math = `movedBookings.length` (not 2N) — `skipOwnerEmail=true` means 1 email per moved booking
- `firstNameOf(fullName)` render helper — first name derived at render time by splitting `booker_name` on whitespace; not stored in DB
- PUSH-10 reason callout: `actor='owner' && reason non-empty` → apology line + "Reason:" block in booker template; LD-07 preserved
- Critical lesson: `booker_first_name` invented in plan body, caught at runtime — real column is `booker_name`; future plans must grep migrations before naming DB fields

## Milestone Summary

**Key Decisions:**

- Refuse-send fail-closed across all 7 senders (Phase 31) — v1.1 carve-out removed; fail-closed is the default for all new email paths going forward
- Wipe migration over dual-read CP-03 for legacy date_overrides rows (Phase 32) — production diagnostic confirmed only 3 rows; clean DML wipe simpler than schema column migration; reusable pattern for low-row-count semantic flips
- ABORT-on-diverge race strategy for pushback commit (Phase 33) — contrasts with Phase 32 union approach; order-dependent cascade math cannot tolerate partial IDs
- Per-booking slot step uses each booking's own `duration_minutes` (Phase 33) — correct boundary snapping per booking's own duration
- 1440 sentinel for no-rules days (Phase 33) — prevents false PAST_EOD flags on fully-blocked days
- PUSH-10 reason callout pattern locked — apology + Reason block when actor='owner'; LD-07 booker-neutrality preserved; mirrors cancel email
- `skipOwnerEmail`/`actor` as canonical owner-batch email suppression — reusable across any future owner-batch operation
- `/gsd:audit-milestone` skipped (yolo mode) — phase-level verifiers (8/8 + 8/8 + 6/6 scenarios) covered all must-haves

**Issues Resolved:**

- Vitest `resolve.alias` string-prefix alias was bleeding sub-paths (e.g. `@/lib/email-sender/quota-guard` was misrouted to mock). Fixed in Plan 32-03 to array/regex exact-match form. Green test files raised from 24/31 to 30/31.
- `booker_first_name` column reference in Plan 33-02 (non-existent; real is `booker_name`). Fixed by orchestrator in commit `bba0e18` before human-verify checkpoint.
- PUSH-10 partial non-fulfillment in Plan 33-03 (reason not plumbed to reschedule email template). Closed by orchestrator in commit `2aa9177` after Phase 33 verifier returned `human_needed`.

**Issues Deferred:**

- `/gsd:audit-milestone` not run for v1.6 (yolo mode; phase-level verification covered must-haves). Documented in MILESTONES.md "What's next" section.
- `oldEndAt` placeholder in `retryPushbackEmailAction` = `input.oldStartAt` — safe today (`sendOwner:false` suppresses owner leg); needs threading if owner retry leg ever activates.
- Pre-existing TS errors in test-mock files (`tests/bookings-api.test.ts` and 8 others referencing removed `__mockSendCalls`/`__setTurnstileResult` helpers). The one failing test file is `bookings-api.test.ts` (fixture mismatch with quota-gated production code; not a regression from v1.6 work).
- Working-tree drift on `.planning/phases/02-owner-auth-and-dashboard-shell/02-VERIFICATION.md` — pre-existing, not staged or resolved during v1.6.
- `slot-picker.tsx` on disk (Andrew Option A from Plan 30-01) — date+slot UI duplicated in `booking-shell.tsx` + `slot-picker.tsx`; resolve when reschedule UI is redesigned.

**Technical Debt Incurred:**

- `firstNameOf(fullName)` render-time split pattern established across Phase 33 surfaces — not stored in DB. If first-name-only is ever needed in a server-side context (e.g. email), the function would need to be moved to `lib/` or the DB schema would need a separate column.
- `skipOwnerEmail`/`actor` pattern now in both `cancelBooking` and `rescheduleBooking` — consistent, but future callers must remember to set both flags for owner-batch paths (easy to forget `actor` even if `skipOwnerEmail` is set; consider a single `batchOwner: true` convenience flag in a future cleanup).
