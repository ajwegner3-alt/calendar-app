## Overview

**Architectural pivots (both shipped to production during verification):**

1. **Direct-Google OAuth replaces Supabase `linkIdentity` for Gmail-connect (commit `ab02a23`)** — `connectGmailAction` now builds the Google auth URL itself and `/auth/gmail-connect/callback` exchanges the code at `oauth2.googleapis.com/token` directly, then writes the encrypted refresh token to `account_oauth_credentials`. Reason: Supabase's `linkIdentity` silently dropped `provider_refresh_token` under several conditions and exposed Supabase's domain on Google's `/permissions` page. `signInWithOAuth` (signup/login) keeps `/auth/google-callback` because it still needs Supabase to create `auth.users` rows. State cookie CSRF: `connectGmailAction` writes a 32-byte random hex token into an httpOnly `gmail_connect_state` cookie (10-min maxAge); callback verifies before exchanging code; cookie deleted after consumption.

2. **Gmail REST API replaces SMTP+OAuth2 (commit `cb82b6f`)** — `lib/email-sender/providers/gmail-oauth.ts` POSTs to `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` with a base64url-encoded RFC-822 message and `Authorization: Bearer <accessToken>`. Reason: the `gmail.send` scope only authorizes the Gmail REST endpoint; SMTP relay needs the much broader `https://mail.google.com/` scope, which we don't request. The pitfall surfaced: nodemailer's XOAUTH2 SMTP handshake silently accepted the wrong-scope token AND returned a synthetic messageId, so callers thought the send succeeded — but Gmail dropped every message after acceptance. Symptom: `bookings.confirmation_email_sent=true` + `email_send_log` rows logged + zero recipient delivery.

See `.planning/phases/35-per-account-gmail-oauth-send/35-DEVIATION-DIRECT-OAUTH.md` for the full pivot post-mortem.

Key decisions:
- Per-account quota isolation via `email_send_log.account_id` filter; warn dedup key is `${today}:${accountId}` (account A's 80% threshold doesn't suppress account B's warning)
- `getSenderForAccount` fail-closed contract: factory never throws; every error path returns a `refusedSender` whose `.send()` resolves `{ success: false, error: "oauth_send_refused: ..." }`. Plan 04 callers branch on `result.success` only; no try/catch needed
- `invalid_grant` is the ONLY DB-write error path — only Google's authoritative revocation triggers `UPDATE account_oauth_credentials SET status='needs_reconnect'`
- `REFUSED_SEND_ERROR_PREFIX = "oauth_send_refused"` exported constant for caller branch logic
- Hosted Supabase migrations applied via `mcp__claude_ai_Supabase__apply_migration` — local `supabase/config.toml` flags only apply to local Supabase
- Nil UUID sentinel `"00000000-0000-0000-0000-000000000000"` for system-level sends (signup, welcome-email) where no per-account context exists

Key commits: `ab32d3e`, `f43cf2c`, `0aaa2d7`, `e09f019`, `f639f0c`, `e427e52`, `e3a7dfb`, `c816e8c`, `66f47f0`, `c9f2312`, `f0da49e`, `03c0ebd`, `ab02a23`, `cb82b6f`, `31db425`, `138cfb0`, `6aecfbb`

**Completed:** 2026-05-08 — Verifier 19/19 PASS; Andrew live-verified A/B/C/D against production (`booking.nsintegrations.com`).

**Details:**

Success Criteria (all met):
1. On the existing `/app/login` card, a user can enter their email and request a magic-link login email without leaving the page or navigating to a separate route. ✓
2. Submitting a known email address and submitting an unknown email address return identical HTTP status codes and identical response body text — no enumeration leakage. ✓ (verified two ways: UI parity live + Supabase MCP confirmed zero `auth.users` and zero `auth.one_time_tokens` rows for unknown email)
3. More than 5 magic-link requests from the same (IP, email) pair within one hour are silently throttled (rate-limited via `rate_limit_events`); throttled requests return the SAME success-shape response as real sends. ✓

Key decisions:
- Silent rate-limit return for enumeration-sensitive endpoints: `requestMagicLinkAction` returns `{ success: true }` on rate-limit miss (NOT `{ formError }`). Throttle status is itself enumeration-sensitive on magic-link
- 5xx-only formError gating for enumeration-safe Supabase actions: only `error.status >= 500 || !error.status` surfaces `{ formError }`. 4xx (including the canonical "unknown email" 400 from `signInWithOtp` with `shouldCreateUser:false`) always logged + swallowed. Mirrors `loginAction` LD pattern (auth-js `error.code` is unreliable; gate on `status` only)
- `shouldCreateUser:false` is the login-only switch — `signInWithOtp` defaults to `true` which silently auto-registers any unknown email. Every magic-link login call site MUST pass `{ options: { shouldCreateUser: false } }`
- Radix Tabs unmount-resets-state: switching tabs unmounts inactive panel, discarding local state and `useActionState` — free state cleanup on tab switch, no `key` prop needed
- Cooldown-on-mount for resend components: `MagicLinkSuccess` starts at 30 (parent only mounts AFTER initial successful send); `ResendVerificationButton` starts at 0 (standalone page where user may land via email link)
- **Four-way enumeration-safety ambiguity invariant** (Plan 02 LD): user-visible response is byte-identical across FOUR distinct backend states — (a) unknown email, (b) our 5/hour rate-limit bucket exhausted, (c) Supabase's internal per-email OTP cooldown active (~60s), (d) genuinely successful send. Direct consequence of the 5xx-only formError gate

**Deviations during verification (non-blocking, captured in 38-03-SUMMARY.md):**

- **Hosted Supabase Site URL was on default `http://localhost:3000`** — Andrew updated to `https://booking.nsintegrations.com` mid-verification. Was not in the original Task 2 dashboard checklist because Phase 34 OAuth uses redirect URL allowlist, not `{{ .SiteURL }}` template interpolation. **Lesson:** future phases that ship hosted email templates with `{{ .SiteURL }}` must include "Verify Site URL field is set to production domain" in their USER-SETUP checklist, AND should include "click the link in the actual email" as a verification step (the bug only manifests in the live email body, not in the dashboard UI).
- **Supabase's internal per-email OTP cooldown (~60s)** observed live: 5 hits to our `auth:magicLink:${ip}:${email}` bucket across 23 minutes resulted in only 2 actual email deliveries because Supabase's cooldown fired faster than our 5/hour bucket on rapid retries. Treated as feature, not bug — strengthens AUTH-29 beyond must-have via the four-way ambiguity invariant.

**Key Decisions:**

- **5xx-only formError gate for enumeration-safe Supabase actions (Phase 38 LD)** — only surface `{ formError }` when `error.status >= 500 || !error.status`; 4xx (including canonical "unknown email" 400) always logged + swallowed. Direct cause of the four-way enumeration-safety ambiguity invariant.
- **Direct-Google OAuth instead of Supabase `linkIdentity` for Gmail-connect (Phase 35 deviation)** — `linkIdentity` silently dropped `provider_refresh_token` under several conditions; direct OAuth callback at `/auth/gmail-connect/callback` is the canonical token-capture path.
- **Gmail REST API instead of nodemailer SMTP+OAuth2 (Phase 35 deviation)** — `gmail.send` scope only authorizes the REST endpoint; SMTP relay needs broader `https://mail.google.com/` scope. Nodemailer's XOAUTH2 SMTP handshake silently accepted wrong-scope tokens AND returned a synthetic messageId — Gmail dropped every message after acceptance (`bookings.confirmation_email_sent=true` + zero recipient delivery).
- **AES-256-GCM at rest for refresh tokens (Phase 34)** — `iv:authTag:ciphertext` lowercase-hex blob; `getKey()` lazy env-var read inside function body for test isolation. Pattern reusable for any future encrypted-blob-at-rest field.
- **Lazy env-var read pattern (Phases 34/35/36/37)** — `getKey()`, `fetchGoogleAccessToken()`, `createResendClient()` all read env vars inside function body, not at module top level. Required for Vitest `beforeEach` env mutation; reusable for all env-var-gated server utilities.
- **`getSenderForAccount` factory routing (Phase 35)** — fail-closed contract; factory never throws; every error path returns `refusedSender` resolving `{ success: false, error: "oauth_send_refused: ..." }`. Plan 04 callers branch on `result.success` only; no try/catch.
