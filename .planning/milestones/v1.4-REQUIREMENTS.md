### Slot Correctness (Headline)

- [x] **SLOT-01**: A booker cannot insert a confirmed booking that overlaps any other confirmed booking on the same `account_id`, regardless of `event_type_id`. DB-enforced via `EXCLUDE USING gist (account_id WITH =, event_type_id WITH <>, during WITH &&) WHERE (status='confirmed')` constraint on `bookings` table. Verified by unit test asserting `error.code === '23P01'` on second INSERT. — SHIPPED via Phase 27 Plan 27-01 (constraint live in production) + Plan 27-03 Test 1.
- [x] **SLOT-02**: SLOT-01 must coexist with v1.1 group-booking capacity. The `event_type_id WITH <>` clause makes this coexist (constraint only fires when event_type_ids differ). Verified by test: insert 3 bookings on a `max_bookings_per_slot=3` event type at same `start_at` → all 3 succeed. — SHIPPED via Phase 27 Plan 27-03 Test 2.
- [x] **SLOT-03**: SLOT-01 must coexist with reschedule semantics. A booking moved via `lib/bookings/reschedule.ts` UPDATE-in-place does NOT block its own new slot. Verified by test exercising the route at the lib layer. — SHIPPED via Phase 27 Plan 27-02 (23P01 → `slot_taken` mapping) + Plan 27-03 Test 5.
- [x] **SLOT-04**: Same-event-type double-booking RCA documented in Phase 27 SUMMARY. The existing `bookings_capacity_slot_idx` partial unique index continues to handle same-event-type race-safety; no regression vs. v1.1 found. — SHIPPED via Phase 27 SUMMARY (rollup section); existing v1.1 mechanism unchanged.
- [x] **SLOT-05**: Production smoke test confirms a cross-event-type collision attempt is rejected. Andrew live-verified on production deploy. — SHIPPED via Phase 27 Plan 27-03 Andrew smoke checkpoint (Phase A passed on `nsi`; Phase B Turnstile-blocked but acceptable per plan).

## Out of Scope

Explicitly excluded from v1.4 to prevent scope creep:

| Feature | Reason |
|---------|--------|
| Multi-host scheduling (two contractors, one calendar) | Not core to solo trade contractor product. Cal.com Teams territory. |
| Round-robin event types | Anti-feature for v1; targets enterprises, not solo contractors. |
| Resource booking (room/equipment shared across event types) | Solo contractor model — the contractor is the resource. |
| Working-hours conflict detection across separate calendars | The contractor's calendar IS the source of truth; no external calendar sync (per v1.0 decision). |
| Auto-cancel existing bookings on date_override insertion | Existing behavior (admin sees + decides) is correct per Features research. |
| Configurable per-event-type buffer asymmetry | Buffer is account-level today; deferring per-event-type buffer to a future milestone. |
| "This conflicts" preview UX banner before submit | The 409 inline-banner pattern from v1.0/v1.1 is sufficient. |
| EXCLUDE constraint with `tstzrange` materialized via function-based index expression | Stack research confirmed only the stored generated column form works. |
| Trigger-based enforcement (Option B) | Option A `EXCLUDE USING gist` with `WITH <>` was the locked primary. Option B was fallback only and never needed. |
| Application-layer SELECT FOR UPDATE conflict check | Bypassable via direct POST; trigger/EXCLUDE is the correct DB-layer enforcement. |
| Two-step `NOT VALID` + `VALIDATE CONSTRAINT` migration | Pre-flight diagnostic showed bookings_total = 6 << 10k threshold; single-step `ADD CONSTRAINT` chosen. |
| Touching `components/ui/calendar.tsx`, `globals.css --color-accent`, or shared shadcn components | v1.3 invariant — per-instance className overrides only. Preserved through v1.4. |
| Removing "Powered by North Star Integrations" footer from public booking surfaces | Public surfaces still ship NSI footer (per v1.2 decision); only auth surfaces lost the pill. |
