# Milestone v1.0: MVP

**Status:** SHIPPED 2026-04-27
**Phases:** 1-9
**Total Plans:** 52

## Overview

A multi-tenant Calendly-style booking tool for trade contractors. v1 ships Andrew's single NSI account end-to-end (schema is multi-tenant from day one; signup/onboarding UI is deferred to v2). Phases follow the dependency chain identified in research: foundation → auth → what-can-be-booked (event types) → when-it's-bookable (availability engine) → the booking transaction (public flow + email + .ics) → lifecycle (cancel/reschedule) → distribution (widget + branding) → hardening + reminders + bookings dashboard list → manual QA sign-off.

Critical pitfalls (double-booking race, timezone/DST, service-role misuse, RLS gaps) were front-loaded into Phase 1 so later phases never re-opened foundational decisions.

**Parallelization actually used:**

- Phases 1 → 2 → 3 sequential.
- Phase 4 (Availability Engine) ran in parallel with Phase 3 once Phase 1 landed.
- Phase 5 ran after Phase 3 + Phase 4 finished.
- Phases 6, 7, 8 ran in parallel after Phase 5.
- Phase 9 sequentially last.

### Phase 4: Availability Engine

**Goal:** Given an event type and date range, the system returns the correct list of bookable UTC slots, accounting for rules, overrides, buffers, notice windows, daily caps, existing bookings, and DST transitions.

**Depends on:** Phase 1 (ran in parallel with Phase 3).

**Plans:** 6

- [x] 04-01: deps-and-accounts-migration — date-fns v4 + @date-fns/tz v1 + shadcn calendar; 4 settings columns on accounts
- [x] 04-02: slot-engine-and-dst-tests — Pure `computeSlots()` + Vitest tests covering March 8 + Nov 1 2026 DST
- [x] 04-03: data-layer-and-server-actions — Zod schemas + server-only loader + 4 Server Actions
- [x] 04-04: weekly-editor-and-settings-panel — Calendly-style per-row editor + Copy-from menu + global settings
- [x] 04-05: date-overrides-ui — Calendar with red/blue dot markers + list view + override modal (Block / Custom hours)
- [x] 04-06: slots-api-route-and-integration-test — GET `/api/slots` route handler + 13-test integration suite

**Details:**

Verified 2026-04-25. `date-fns@4.1.0` + `@date-fns/tz@1.4.1`; **DO NOT** install `date-fns-tz` (the v2/v3-era package). `formatInTimeZone` does NOT exist in `@date-fns/tz`; use `TZDate` and `tz()` instead. Critical DST learning: `addMinutes(midnight, startMinute)` is DST-UNSAFE for window endpoints — adds elapsed UTC milliseconds, not wall-clock minutes. Fix: `new TZDate(y, m-1, d, Math.floor(min/60), min%60, 0, TZ)`. `TZDate.toISOString()` returns offset format, not UTC Z; use `new Date(tzDate.getTime()).toISOString()` for UTC Z output. Buffer-overlap removes upstream adjacent slot (with buffer=15min and booking 10:00-10:30, slots 9:30, 10:00, AND 10:30 are removed). `computeSlots()` is pure: NO `new Date()` inside; `now` MUST be injected via `SlotInput.now`. Account settings columns: `buffer_minutes`, `min_notice_hours`, `max_advance_days`, `daily_cap` (NULL = no cap). Empty windows = closed weekday (no `is_open` column; presence/absence of rows IS the open/closed state). Date override mutual exclusion enforced in action layer (always delete-then-insert). Mon-first display order in weekly editor. server-only Vitest alias added to `vitest.config.ts` for route-handler integration tests.

## Milestone Summary

**Decimal Phases:** None — milestone executed exactly as planned with no insertions.

**Key Decisions:**

- Calendly-style booking tool, not a general calendar — narrower scope ships faster.
- Multi-tenant architecture from day one, single tenant in v1 — supports the NSI business model (sell to clients) without forcing a later data migration; signup UI deferred.
- Supabase is the sole source of truth — no Google Calendar sync (explicit user preference; removes OAuth/sync complexity).
- DB-level race-safe booking via `bookings_no_double_book` partial unique index — authoritative double-book guard; runtime-proved in Vitest race test.
- `timestamptz` everywhere + IANA TZ strings + `date-fns v4 + @date-fns/tz` (no raw Date math).
- Service-role gate via `import "server-only"` in `lib/supabase/admin.ts` line 1.
- Vendor `@nsi/email-sender` into `lib/email-sender/` (NOT `npm install ../email-sender`) — Vercel build cannot resolve sibling-relative `file:../`.
- Gmail SMTP via owner's personal Gmail (post-Resend pivot) for transactional delivery — `lib/email-sender/providers/gmail.ts`.
- POST `/api/bookings` is a Route Handler not a Server Action — Server Actions cannot return 409.
- Direct-call Server Action contract for forms (NOT FormData) — avoids serialize-nested-arrays pitfall for `custom_questions`/availability windows.
- CSP lives ONLY in `proxy.ts` — `next.config.ts` cannot conditionally delete a header at runtime.
- `X-Frame-Options` layering: `next.config.ts` global SAMEORIGIN + `proxy.ts` deletes on `/embed/*`.
- `nsi-booking:height` postMessage protocol locked for embed widget auto-resize.
- Token rotation on every reminder send; reminder retry on send failure = NONE by design (RESEARCH Pitfall 4).
- Vercel Pro tier required for hourly cron (`vercel.json` `0 * * * *` doesn't deploy on Hobby).
- Per-account branding = logo + colors only (no custom CSS) — support burden too high for v1 value.
- Free bookings only; no Stripe — trade contractors don't charge for quote consultations.

**Issues Resolved:**

- Race condition between concurrent booking submits → DB-level partial unique index (Phase 1).
- DST window-endpoint bug (`addMinutes(midnight, n)` adds elapsed UTC ms, not wall-clock minutes) → `TZDate(y, m-1, d, hh, mm, 0, TZ)` constructor pattern (Phase 4).
- `formatInTimeZone` does not exist in `@date-fns/tz` v1 → use `TZDate` + `tz()` (Phase 4).
- Server Actions cannot return 409 → POST `/api/bookings` Route Handler (Phase 5).
- FormData cannot serialize nested arrays for `custom_questions` → direct-call action contract (Phase 3).
- TooltipProvider not bundled inside SidebarProvider in installed shadcn version → wrapper added in shell layout (Phase 2).
- `next.config.ts` cannot conditionally delete `X-Frame-Options` → `proxy.ts` exclusive CSP/XFO ownership (Phase 7).
- Concurrent same-token reschedule could rotate to different new tokens → double CAS guard (Phase 6).
- Reminder cron duplicate invocation could send two reminders → claim-once via compare-and-set UPDATE on `reminder_sent_at` (Phase 8).
- Multi-agent parallel-wave commits swept in untracked sibling files → noted as technical debt; serialize commits or per-agent worktrees recommended for future YOLO runs (Phase 8).
