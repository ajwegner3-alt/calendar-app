## Overview

v1.2 unified the entire owner-facing app under a single North Star Integrations visual language while preserving each contractor's per-account branding on public booking surfaces and transactional emails. The owner shell, all 7 auth pages, and the 3-step onboarding wizard re-skinned to the lead-scoring "Simple Light" reference (`bg-gray-50` + blue-blot `BackgroundGlow` + glass "NorthStar" header pill + Inter weights 400-800). Public booking pages and the embed widget adopted the same layout pattern with the customer's `brand_primary` driving the glow tint and a customer-branded header pill. The branding editor collapsed from 5 controls to 2 (logo + `brand_primary`); 3 deprecated branding columns + 2 Postgres ENUM types were permanently dropped via a two-step deploy protocol with a mandatory 30-minute Vercel function drain.

**Branding rule (locked across all 8 phases):**

| Surface | Whose colors |
|---|---|
| Owner app (`/app/*`, `/auth/*`, `/onboarding/*`, named auth pages) | NSI ONLY (`#3B82F6` blue + gray-50) |
| Public booking pages + embed | Customer (`brand_primary`) |
| 6 transactional emails | Customer (`brand_primary`) |

**Six locked decisions every phase plan restated in its preamble:**
1. JIT pitfall: runtime hex via `style={{ ... }}` only â€” never `bg-[${color}]` dynamic Tailwind
2. Email strategy: solid-color-only table band â€” no CSS gradients, no VML in email HTML
3. CSP: lives only in `proxy.ts`, never `next.config.ts`
4. Two-stage owner auth: RLS pre-check before service-role mutation
5. `background_shade` ENUM type must be dropped alongside the column in the DROP migration
6. DROP migration = two-step deploy (code-stop-reading deploy, wait 30 min, then DROP SQL)

Phases executed strictly sequentially: **Phase 14 â†’ 15 â†’ 16 â†’ 17 â†’ 18 â†’ 19 â†’ 20 â†’ 21**. Within each phase, plan-level parallelization was enabled (`parallelization=true`); Phase 17 used 5 waves with 5 parallel plans in Wave 3.
