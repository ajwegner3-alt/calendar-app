  <name>Task 1: Create account_oauth_credentials migration</name>
  <files>supabase/migrations/20260506120000_phase34_account_oauth_credentials.sql</files>
  <action>
Create a new forward-only migration following the project's pattern (see `supabase/migrations/20260428120002_phase10_accounts_rls_and_trigger.sql` for header style + RLS conventions). The migration must:

1. Create table `account_oauth_credentials` with columns:
   - `id uuid primary key default gen_random_uuid()`
   - `user_id uuid not null references auth.users(id) on delete cascade`
   - `provider text not null check (provider in ('google'))` — extensible to future providers
   - `refresh_token_encrypted text not null` — AES-256-GCM ciphertext stored as `iv:authTag:ciphertext` hex (Plan 02 produces this format)
   - `granted_scopes text` — space-delimited Google OAuth scope list (e.g. `email profile https://www.googleapis.com/auth/gmail.send`)
   - `status text not null default 'connected' check (status in ('connected', 'needs_reconnect'))` — Phase 35 will flip to 'needs_reconnect' on invalid_grant
   - `connected_at timestamptz not null default now()`
   - `last_refresh_at timestamptz` — Phase 35 updates this on each successful token refresh
   - `created_at timestamptz not null default now()`
   - `updated_at timestamptz not null default now()`
   - `unique (user_id, provider)` — one credential per user per provider

2. Index: `create index account_oauth_credentials_user_id_idx on account_oauth_credentials (user_id);` (supports Phase 35's getSenderForAccount lookup).

3. Enable RLS: `alter table account_oauth_credentials enable row level security;`

4. Add SELECT-only RLS policy `credentials_select_own` for `to authenticated` using `auth.uid() = user_id`. Do NOT add INSERT/UPDATE/DELETE policies — all writes go through the service-role admin client (createAdminClient bypasses RLS by design; this prevents a client-side bug from overwriting refresh_token_encrypted).

5. Add `updated_at` trigger using existing project convention: function `update_oauth_credentials_updated_at` returns trigger that sets `new.updated_at = now()`; trigger `oauth_credentials_updated_at before update on account_oauth_credentials for each row`.

6. Comment block at top of file explaining: phase 34, why writes go through admin client, the encrypted-format spec (iv:authTag:ciphertext hex), and that this is forward-only (no rollback file).

DO NOT seed any rows. DO NOT touch the existing `accounts` table. DO NOT add `email_provider` column — that lands in Phase 35.

After writing, run `npx supabase db reset` locally OR `npx supabase db diff --linked` to verify the migration parses and applies cleanly against the local Supabase instance. (If `npx supabase db reset` is too disruptive in dev, at minimum `psql` or `supabase migration list` to confirm filename ordering is correct.)
