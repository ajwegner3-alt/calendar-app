Refactor `POST /api/stripe/checkout` to accept `{ tier: 'basic' | 'widget' | 'branding', interval: 'monthly' | 'annual' }` instead of just `{ interval }`. Read `app/api/stripe/checkout/route.ts` to understand the current shape. The existing route flow is:
1. Auth check (401 if no session)
2. Account lookup (404 if no row)
3. Parse body → `interval` only (currently)
4. `priceId = PRICES[interval].priceId` (current — being replaced)
5. Production guard: reject `price_placeholder` IDs with `{error:'missing_price_id'}`
6. Stripe customer upsert (creates customer + persists `stripe_customer_id`)
7. `stripe.checkout.sessions.create()` with line_items, success_url, cancel_url, metadata
8. Return 200 + `{ url }`

Changes (surgical — only steps 3 and 4 change):

**Imports section:**
- Replace `import { PRICES, type PriceInterval } from "@/lib/stripe/prices";` (or similar current shape) with:
  ```typescript
  import { getPriceId, type PriceTier, type PriceInterval } from "@/lib/stripe/prices";
  ```
  Note: `PRICES` import may still be needed if the route reads `amountCents` or `label` elsewhere — keep `PRICES` in the import list IF AND ONLY IF the existing code references it after the refactor.

**Body parsing block (~lines 53-73 area, after the auth/account checks):**
- Replace the current body parsing (which extracts only `interval`) with this exact shape:
  ```typescript
  const body = await req.json().catch(() => ({})) as Record<string, unknown>;
  const rawTier = body?.tier;
  const rawInterval = body?.interval;

  // Reject Branding tier immediately — it uses the consult link, not Stripe.
  if (rawTier === "branding") {
    return NextResponse.json(
      { error: "use_consult_link" },
      { status: 400, headers: NO_STORE },
    );
  }

  // Validate tier is one of the two Stripe-backed tiers.
  if (rawTier !== "basic" && rawTier !== "widget") {
    return NextResponse.json(
      { error: "unknown_tier" },
      { status: 400, headers: NO_STORE },
    );
  }
  const tier: PriceTier = rawTier;
  const interval: PriceInterval = rawInterval === "annual" ? "annual" : "monthly";

  const priceId = getPriceId(tier, interval);
  ```

**Existing `missing_price_id` production guard:**
- KEEP it. The guard checks `priceId.startsWith("price_placeholder")` and returns 400. Now it guards against any of the 4 placeholder Price IDs being used in production. No behavior change needed.

**Existing `line_items` parameter in `stripe.checkout.sessions.create()`:**
- KEEP it. `priceId` variable now points to the correct tier-aware Price ID. Stripe call signature is unchanged.

**Metadata on the session create call:**
- The existing call likely has `metadata: { accountId, interval }` or similar. ADD `tier` to the metadata: `metadata: { accountId, tier, interval }`. This is defensive — webhook tier derivation uses `listLineItems` (not metadata) per the verification gate, but having `tier` in metadata is a useful debug breadcrumb. Cost: zero. Risk: zero.

**What MUST NOT change:**
- The auth check (401) — runs BEFORE body parsing
- The account lookup (404) — runs BEFORE body parsing
- The Stripe customer upsert logic + `.is("stripe_customer_id", null)` race guard
- The `NO_STORE` headers helper
- The `success_url` + `cancel_url` shape
- The return shape `{ url: session.url }`
- The error response format (`NextResponse.json({error: '...'}, {status: 400, headers: NO_STORE})`)

**Order of validations matters:** auth (401) → account (404) → branding-reject (400) → unknown-tier-reject (400) → placeholder-price-reject (400 if production) → stripe call.

**Test fixture impact:** If `tests/checkout-route.test.ts` exists, it currently POSTs `{interval: 'monthly'}`. Those tests will need to POST `{tier: 'basic', interval: 'monthly'}` to pass — update test fixtures in this plan as part of this task. (If no test file exists, skip.)
