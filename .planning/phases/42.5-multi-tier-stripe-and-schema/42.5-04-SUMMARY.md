## Code Shape (for UAT planner reference)

The new logic block lives at the tail of `handleCheckoutSessionCompleted`, immediately before `return { accountId };`:

```typescript
// === Phase 42.5: derive plan_tier from line items ===
let lineItems: Stripe.ApiList<Stripe.LineItem>;
try {
  lineItems = await stripe.checkout.sessions.listLineItems(session.id, {
    limit: 1,
    expand: ["data.price"],
  });
} catch (err) {
  console.error("[stripe-webhook] checkout.session.completed: listLineItems failed", {
    sessionId: session.id,
    accountId,
    error: err instanceof Error ? err.message : String(err),
  });
  throw new Error("listLineItems_failed");
}

const priceId = lineItems.data[0]?.price?.id ?? null;
const planTier = priceId ? priceIdToTier(priceId) : null;

if (!planTier) {
  console.warn("[stripe-webhook] checkout.session.completed: could not derive plan_tier", {
    sessionId: session.id,
    priceId,
    accountId,
  });
} else {
  const { error: tierWriteErr } = await admin
    .from("accounts")
    .update({ plan_tier: planTier })
    .eq("id", accountId);

  if (tierWriteErr) {
    console.error("[stripe-webhook] checkout.session.completed: plan_tier write failed", {
      sessionId: session.id,
      accountId,
      planTier,
      error: tierWriteErr.message,
    });
    throw new Error("plan_tier_write_failed");
  }

  console.log("[stripe-webhook] checkout.session.completed: plan_tier set", {
    sessionId: session.id,
    accountId,
    priceId,
    planTier,
  });
}
```

**Log shapes for UAT grep filters:**
- Success: `[stripe-webhook] checkout.session.completed: plan_tier set` (object has `sessionId`, `accountId`, `priceId`, `planTier`)
- Unknown Price: `[stripe-webhook] checkout.session.completed: could not derive plan_tier`
- API failure: `[stripe-webhook] checkout.session.completed: listLineItems failed`
- DB failure: `[stripe-webhook] checkout.session.completed: plan_tier write failed`

## Decisions Made

- **No `.is("plan_tier", null)` guard on the new UPDATE.** Unlike the `stripe_customer_id` write which guards against a cross-event race with `customer.subscription.created`, there is no analogous race for `plan_tier` (only `checkout.session.completed` ever writes it). The UPDATE is naturally idempotent — writing the same `'basic'`/`'widget'` value twice is harmless.
- **Unknown-Price path returns 200, not 500.** Throwing on unknown Price would create a Stripe retry storm whenever a stripe-CLI test fixture or pre-mapped SKU flows through. Logging a clear warning lets the human notice and update `lib/stripe/prices.ts` without paging Stripe's retry machinery. `plan_tier` remains NULL — Phase 42.6 gating treats NULL as trialing (allowed access), so this is the safe default.
- **listLineItems failure path throws.** A genuine Stripe API outage is exactly when retry-with-backoff is the correct response — the `stripe_customer_id` write is idempotent on retry (race guard), so the entire handler is safe to re-run.
