# Phase 42.5: Multi-Tier Stripe + Schema — Research

**Researched:** 2026-05-10
**Domain:** Stripe multi-SKU checkout, Supabase schema migration, Next.js App Router billing UI
**Confidence:** HIGH — all findings verified from live codebase files + Stripe SDK type definitions

---

## Summary

Phase 42.5 refactors the existing Phase 42 plumbing to support three tiers (Basic, Widget, Branding) instead of one. The plumbing files are all on disk and working — none are thrown away. The work is surgical: expand the prices config to a 4-SKU map, add `tier` validation to the checkout route, add a `stripe.checkout.sessions.listLineItems()` call in the webhook to derive `plan_tier`, add `plan_tier` column to the schema, and replace the single `PlanSelectionCard` with a 3-card `TierGrid` layout.

The most important non-obvious finding: **the `checkout.session.completed` webhook event payload does NOT include `line_items` by default.** The field is optional and only appears when expanded via an API retrieve call. To derive `plan_tier` from the Price ID, the webhook handler must call `stripe.checkout.sessions.listLineItems(session.id)` after receiving the event. This is a separate API roundtrip. The ROADMAP verification gate explicitly requires this approach (not metadata, not hardcoding).

The Branding card is a link, not a Stripe checkout — it renders a styled `<a href={process.env.NSI_BRANDING_BOOKING_URL}>` (server-side env var, NOT `NEXT_PUBLIC_` prefixed) and triggers no API calls. Access pattern: server component reads `process.env.NSI_BRANDING_BOOKING_URL` at render time and passes it as a prop to the tier grid.

**Primary recommendation:** Three distinct plans in a new `lib/stripe/prices.ts` shape keyed by `{ tier, interval }`. The checkout route validates `{ tier, interval }` body and rejects `branding` with 400. The webhook calls `listLineItems` to get the price ID, looks it up in a `PRICE_ID_TO_TIER` map exported from `prices.ts`, then writes `plan_tier` alongside existing subscription column writes.

### Pattern 1: 4-SKU prices config shape

The current `lib/stripe/prices.ts` exports a flat `PRICES` object keyed by `{ monthly, annual }`. Phase 42.5 refactors it to a nested object keyed by `{ tier: 'basic' | 'widget', interval: 'monthly' | 'annual' }` plus a reverse-lookup map and helper functions.

```typescript
// lib/stripe/prices.ts — FULL REPLACEMENT
import "server-only";

export type PriceTier = "basic" | "widget";
export type PriceInterval = "monthly" | "annual";

function dollars(cents: number): string {
  return `$${(cents / 100).toFixed(cents % 100 === 0 ? 0 : 2)}`;
}

const basicMonthlyCents = parseInt(process.env.STRIPE_PRICE_BASIC_MONTHLY_CENTS ?? "2900", 10);
const basicAnnualCents  = parseInt(process.env.STRIPE_PRICE_BASIC_ANNUAL_CENTS  ?? "27840", 10);
const widgetMonthlyCents = parseInt(process.env.STRIPE_PRICE_WIDGET_MONTHLY_CENTS ?? "4900", 10);
const widgetAnnualCents  = parseInt(process.env.STRIPE_PRICE_WIDGET_ANNUAL_CENTS  ?? "47040", 10);

export const PRICES = {
  basic: {
    monthly: {
      priceId: process.env.STRIPE_PRICE_ID_BASIC_MONTHLY ?? "price_placeholder_basic_monthly",
      amountCents: basicMonthlyCents,
      label: `${dollars(basicMonthlyCents)}/month`,
    },
    annual: {
      priceId: process.env.STRIPE_PRICE_ID_BASIC_ANNUAL ?? "price_placeholder_basic_annual",
      amountCents: basicAnnualCents,
      totalLabel: `${dollars(basicAnnualCents)}/year`,
      monthlyEquivalentLabel: `${dollars(Math.round(basicAnnualCents / 12))}/month`,
      savingsPct: Math.max(0, Math.round((1 - basicAnnualCents / (basicMonthlyCents * 12)) * 100)),
    },
  },
  widget: {
    monthly: {
      priceId: process.env.STRIPE_PRICE_ID_WIDGET_MONTHLY ?? "price_placeholder_widget_monthly",
      amountCents: widgetMonthlyCents,
      label: `${dollars(widgetMonthlyCents)}/month`,
    },
    annual: {
      priceId: process.env.STRIPE_PRICE_ID_WIDGET_ANNUAL ?? "price_placeholder_widget_annual",
      amountCents: widgetAnnualCents,
      totalLabel: `${dollars(widgetAnnualCents)}/year`,
      monthlyEquivalentLabel: `${dollars(Math.round(widgetAnnualCents / 12))}/month`,
      savingsPct: Math.max(0, Math.round((1 - widgetAnnualCents / (widgetMonthlyCents * 12)) * 100)),
    },
  },
} as const;

/**
 * Reverse-lookup map: Stripe Price ID → plan_tier value.
 * Used by webhook handleCheckoutSessionCompleted to derive plan_tier
 * from session.line_items[0].price.id (verification gate requirement).
 * Must not hardcode tier values; derives them from the env-var-driven PRICES map.
 */
export const PRICE_ID_TO_TIER: Record<string, PriceTier> = {
  [PRICES.basic.monthly.priceId]:  "basic",
  [PRICES.basic.annual.priceId]:   "basic",
  [PRICES.widget.monthly.priceId]: "widget",
  [PRICES.widget.annual.priceId]:  "widget",
};

/**
 * Helper: resolve Price ID from tier + interval.
 * Returns the price ID or throws if unknown tier/interval combination.
 */
export function getPriceId(tier: PriceTier, interval: PriceInterval): string {
  return PRICES[tier][interval].priceId;
}

/**
 * Helper: derive plan_tier from a Price ID.
 * Returns null if the Price ID is not in the 4-SKU map (unknown/legacy price).
 */
export function priceIdToTier(priceId: string): PriceTier | null {
  return PRICE_ID_TO_TIER[priceId] ?? null;
}
```

**Key design decisions:**
- Placeholder Price IDs must be distinct per SKU so PRICE_ID_TO_TIER lookup doesn't accidentally alias them. Each placeholder must be unique (e.g., `price_placeholder_basic_monthly`).
- The `priceIdToTier` function returns `null` (not throws) for unknown IDs — lets the webhook log a warning and skip writing `plan_tier` rather than 500-ing.
- The old `STRIPE_PRICE_ID_MONTHLY`/`STRIPE_PRICE_ID_ANNUAL` env vars are gone. The checkout route import changes from `PRICES[interval]` to `getPriceId(tier, interval)`.

### Pattern 2: Checkout route body validation

Current body: `{ interval: 'monthly' | 'annual' }` — single implied tier.
New body: `{ tier: 'basic' | 'widget' | 'branding', interval: 'monthly' | 'annual' }`.

```typescript
// In POST /api/stripe/checkout (route.ts)
const body = await req.json().catch(() => ({})) as Record<string, unknown>;
const rawTier = body?.tier;
const rawInterval = body?.interval;

// Reject branding immediately — no Stripe checkout for it
if (rawTier === "branding") {
  return NextResponse.json({ error: "use_consult_link" }, { status: 400, headers: NO_STORE });
}

// Validate tier
if (rawTier !== "basic" && rawTier !== "widget") {
  return NextResponse.json({ error: "unknown_tier" }, { status: 400, headers: NO_STORE });
}
const tier: PriceTier = rawTier;
const interval: PriceInterval = rawInterval === "annual" ? "annual" : "monthly";

const priceId = getPriceId(tier, interval);
```

The rest of the route (customer upsert, session create, etc.) is unchanged. The `session.create` call passes `priceId` exactly as before.

**Error response table:**
| Condition | Status | Body |
|-----------|--------|------|
| `tier === 'branding'` | 400 | `{ error: 'use_consult_link' }` |
| Unknown/missing tier | 400 | `{ error: 'unknown_tier' }` |
| Placeholder price in production | 400 | `{ error: 'missing_price_id' }` (existing guard) |
| No auth session | 401 | `{ error: 'unauthorized' }` |
| No account row | 404 | `{ error: 'account_not_found' }` |

### Pattern 3: Webhook tier derivation via listLineItems

**Critical finding (HIGH confidence):** The `checkout.session.completed` event payload does NOT include `line_items` by default. The field is typed as `line_items?: ApiList<LineItem>` in the Stripe SDK — the `?` means it is absent from webhook payloads. Stripe's official fulfillment docs confirm you must call `stripe.checkout.sessions.listLineItems(session.id)` (or `stripe.checkout.sessions.retrieve(session.id, { expand: ['line_items'] })`) to access them.

The ROADMAP verification gate says: "webhook resolves `plan_tier` by matching `session.line_items[0].price.id` against the 4-SKU map in `lib/stripe/prices.ts` (not by reading metadata, not by hardcoding)." This requires the API call approach.

```typescript
// In handleCheckoutSessionCompleted (webhook route)
// AFTER writing stripe_customer_id (existing logic), ALSO derive + write plan_tier:

const lineItems = await stripe.checkout.sessions.listLineItems(session.id, {
  limit: 1,
  expand: ["data.price"],
});

const priceId = lineItems.data[0]?.price?.id ?? null;
const planTier = priceId ? priceIdToTier(priceId) : null;

if (!planTier) {
  // Unknown price ID or no line items — log warning, don't write plan_tier.
  // This is non-fatal: the subscription will still be active, but plan_tier
  // will remain NULL (Phase 42.6 gating treats NULL as trialing — allowed access).
  console.warn("[stripe-webhook] checkout.session.completed: could not derive plan_tier", {
    sessionId: session.id,
    priceId,
    accountId,
  });
} else {
  const { error: tierWriteErr } = await admin
    .from("accounts")
    .update({ plan_tier: planTier })
    .eq("id", accountId);

  if (tierWriteErr) {
    // Throw so the outer try/catch rolls back the dedupe row (webhook retries cleanly)
    throw new Error("plan_tier_write_failed");
  }
}
```

**Idempotency:** The `stripe_webhook_events` dedupe table prevents re-processing. If `plan_tier` is already set (duplicate delivery somehow bypasses dedupe), the UPDATE is idempotent — writing the same value again is harmless. No `.is("plan_tier", null)` guard needed (unlike `stripe_customer_id` which guards against race).

**Ordering:** The `listLineItems` call happens after the `stripe_customer_id` write. If the API call fails (Stripe is down), we throw so the dedupe row is rolled back and Stripe retries the entire event — the customer_id write will be a no-op on retry (idempotent conditional update).

**Latency:** One extra Stripe API call per first-checkout event. Acceptable — this is an async webhook handler, not a user-facing request.

### Pattern 4: plan_tier schema migration

```sql
-- supabase/migrations/YYYYMMDDHHMMSS_phase42_5_plan_tier.sql

BEGIN;

ALTER TABLE public.accounts
  ADD COLUMN plan_tier TEXT
    CHECK (plan_tier IS NULL OR plan_tier IN ('basic', 'widget'));

COMMENT ON COLUMN public.accounts.plan_tier IS
  'Stripe subscription tier. Written by webhook on checkout.session.completed
   by matching the purchased Price ID against the 4-SKU map. NULL = trialing
   (before first checkout) or grandfathered v1.7 accounts. Phase 42.6 reads
   this for widget feature gating.';

-- Backfill: all existing accounts get NULL (per SC-1).
-- The DEFAULT IS NULL, so no explicit backfill UPDATE needed.
-- Existing rows default to NULL automatically.

COMMIT;
```

**No UPDATE backfill needed** — Postgres `ALTER TABLE ADD COLUMN` with no `DEFAULT` clause leaves existing rows with `NULL` in the new column. This is exactly what SC-1 requires ("all existing v1.7 grandfathered accounts have `plan_tier = NULL`").

**Trigger compatibility:** The `provision_account_for_new_user` function (Phase 41 migration) does NOT INSERT `plan_tier`. Since the column defaults to NULL, new signups will automatically get `plan_tier = NULL` — correct behavior per SC-1's "NULL allowed (trialing accounts before first checkout)". No trigger change needed.

**RLS:** No RLS change. The existing "owners read own account" policy allows SELECT on all columns of the owner's row. The webhook uses the admin client (service role) which bypasses RLS entirely.

**Migration naming convention:** Use `20260510HHMMSS_phase42_5_plan_tier.sql`. Pick a timestamp later than the Phase 41 migration (`20260510120000`) — e.g., `20260510130000`.

### Pattern 5: TierGrid UI layout

The current billing page renders a single `PlanSelectionCard`. Phase 42.5 replaces it with a `TierGrid` component containing three `TierCard` children.

**Layout approach:** `grid grid-cols-1 md:grid-cols-3 gap-6` — stacked on mobile, 3-column at md breakpoint (768px). The verification gate requires all three cards visible without horizontal scroll at 1024px viewport — `md:grid-cols-3` achieves this since 1024px > 768px (md).

**Card variants:**
1. **SubscribeTierCard** (Basic, Widget) — renders Tabs toggle (monthly/annual) + price display + Subscribe button (POSTs to checkout route). Passes `{ tier, interval }` in the fetch body.
2. **ConsultTierCard** (Branding) — renders a static description + a link button styled distinctly. No toggle, no POST.

**"Most popular" badge:** Use `<Badge variant="default">Most popular</Badge>` in the Widget card's `CardHeader` — the `Badge` component already supports this variant (confirmed from `badge.tsx`).

**File structure recommendation:**

```
app/(shell)/app/billing/_components/
├── tier-grid.tsx          # "use client" — manages global toggle state, renders 3 TierCard children
├── subscribe-tier-card.tsx # "use client" — Basic or Widget card with Subscribe logic
└── consult-tier-card.tsx  # Server or client component — static link card for Branding
```

Alternatively: one `tier-grid.tsx` that handles all three cards inline. The planner decides granularity — research documents both as viable.

**Global vs per-card toggle:** The ROADMAP says "each Stripe card with a monthly/annual toggle (or one global toggle — designer's call; both are CONTEXT-compatible)." Recommendation: **one global toggle** (simpler state management, single source of truth for interval, cleaner code). The toggle lives in `TierGrid` and passes `interval` down to each `SubscribeTierCard`. This is the more common SaaS pattern (see Stripe's own pricing page, GitHub Copilot, etc.).

```typescript
// tier-grid.tsx — "use client"
"use client";
import { useState } from "react";
import { Tabs, TabsList, TabsTrigger } from "@/components/ui/tabs";
import { Badge } from "@/components/ui/badge";
import { SubscribeTierCard } from "./subscribe-tier-card";
import { ConsultTierCard } from "./consult-tier-card";
import type { PriceTier, PriceInterval } from "@/lib/stripe/prices";

interface TierGridProps {
  basicMonthlyLabel: string;
  basicAnnualMonthlyEquivalentLabel: string;
  basicAnnualTotalLabel: string;
  basicSavingsPct: number;
  widgetMonthlyLabel: string;
  widgetAnnualMonthlyEquivalentLabel: string;
  widgetAnnualTotalLabel: string;
  widgetSavingsPct: number;
  brandingBookingUrl: string;   // from process.env.NSI_BRANDING_BOOKING_URL (server-passed prop)
}

export function TierGrid({
  basicMonthlyLabel, basicAnnualMonthlyEquivalentLabel, basicAnnualTotalLabel, basicSavingsPct,
  widgetMonthlyLabel, widgetAnnualMonthlyEquivalentLabel, widgetAnnualTotalLabel, widgetSavingsPct,
  brandingBookingUrl,
}: TierGridProps) {
  const [interval, setInterval] = useState<PriceInterval>("annual");

  return (
    <div className="space-y-4">
      {/* Global toggle */}
      <div className="flex justify-center">
        <Tabs value={interval} onValueChange={(v) => setInterval(v as PriceInterval)}>
          <TabsList>
            <TabsTrigger value="monthly">Monthly</TabsTrigger>
            <TabsTrigger value="annual" className="gap-2">
              Annual
              <Badge variant="secondary">Save up to {Math.max(basicSavingsPct, widgetSavingsPct)}%</Badge>
            </TabsTrigger>
          </TabsList>
        </Tabs>
      </div>

      {/* Card grid */}
      <div className="grid grid-cols-1 md:grid-cols-3 gap-6">
        <SubscribeTierCard tier="basic" interval={interval} monthlyLabel={basicMonthlyLabel}
          annualMonthlyEquivalentLabel={basicAnnualMonthlyEquivalentLabel}
          annualTotalLabel={basicAnnualTotalLabel} savingsPct={basicSavingsPct} />
        <SubscribeTierCard tier="widget" interval={interval} monthlyLabel={widgetMonthlyLabel}
          annualMonthlyEquivalentLabel={widgetAnnualMonthlyEquivalentLabel}
          annualTotalLabel={widgetAnnualTotalLabel} savingsPct={widgetSavingsPct}
          featured />
        <ConsultTierCard bookingUrl={brandingBookingUrl} />
      </div>
    </div>
  );
}
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tier inference in webhook | `if/else` on metadata or hardcoded strings | `priceIdToTier()` from `lib/stripe/prices.ts` | Verification gate explicitly forbids metadata/hardcoding; map keeps single source of truth |
| Line items retrieval | Trusting `session.line_items` from webhook payload | `stripe.checkout.sessions.listLineItems(session.id)` | Field is NOT in webhook payloads — always absent unless explicitly fetched |
| "Most popular" badge | Custom highlight div | `<Badge variant="default">` (already installed) | Consistent with existing badge usage in PlanSelectionCard |
| 3-column layout | CSS grid with pixel-based breakpoints | Tailwind `grid-cols-1 md:grid-cols-3 gap-6` | Matches existing Tailwind-only pattern; no custom CSS |
| Branding card link button | Custom anchor styling | `<Button asChild><a href={...}>` | Uses the existing button component's `asChild` prop for semantic correctness |
| Plan tier validation | Complex switch/case | Simple string union check | Two valid values (`'basic'`, `'widget'`); TypeScript narrowing handles the rest |

## Common Pitfalls

### Pitfall 1: Assuming line_items is in the webhook event payload

**What goes wrong:** Code reads `session.line_items?.data[0]?.price?.id` from the webhook event object and gets `undefined`. `priceIdToTier(undefined)` returns `null`. `plan_tier` is never written.

**Why it happens:** Stripe's webhook delivery sends the Session object in its "minimal form" — `line_items` is an expandable (includable) field and is absent unless explicitly fetched via the API.

**SDK evidence:** `line_items?: ApiList<LineItem>` — the `?` confirms optionality. The SDK docs string says "includable" not "included."

**How to avoid:** Call `stripe.checkout.sessions.listLineItems(session.id, { limit: 1, expand: ["data.price"] })` after receiving the event. This is one extra Stripe API call; it is acceptable in a webhook handler.

**Warning signs:** `plan_tier` remains NULL after checkout completion even when the webhook received a 200.

### Pitfall 2: PRICE_ID_TO_TIER map built with placeholder Price IDs

**What goes wrong:** In development without real Price IDs set, `PRICE_ID_TO_TIER` is built from placeholder strings like `"price_placeholder_basic_monthly"`. If the checkout route creates a session with this placeholder Price ID (allowed in non-production), the webhook would receive `price_placeholder_basic_monthly` as the price ID and look it up in the map. This works — but ONLY if the placeholder keys are distinct.

**Why it happens:** Two placeholders with the same value (e.g., both `"price_placeholder"`) would alias them in the reverse lookup, causing ambiguous tier resolution.

**How to avoid:** Ensure each of the four placeholder values is unique. The pattern `"price_placeholder_{tier}_{interval}"` prevents aliasing. Verify with a unit assertion or just by inspection.

### Pitfall 3: `plan_tier` column write in wrong handler

**What goes wrong:** `plan_tier` is written in `handleSubscriptionEvent` (customer.subscription.created) instead of `handleCheckoutSessionCompleted`. The subscription event doesn't have easy access to the Price ID that was purchased (it's in the subscription items, but resolving it reliably requires care with the 2026-04-22.dahlia API version field path).

**Why it happens:** It seems logical to write billing columns in the subscription event.

**How to avoid:** Write `plan_tier` in `handleCheckoutSessionCompleted` only. That's where the Price ID is available via `listLineItems`. The subscription handler (`handleSubscriptionEvent`) already writes `plan_interval`, `stripe_subscription_id`, `current_period_end`, and `subscription_status` — adding `plan_tier` there would require additional API calls and duplicate the listLineItems pattern. Keep the concerns separate.

**Alternative if planners disagree:** `plan_tier` could also be derived from `sub.items.data[0]?.price?.id` in `handleSubscriptionEvent` (the subscription object includes price) — but the `checkout.session.completed` approach is cleaner for a first-checkout flow and the ROADMAP verification gate names that handler explicitly.

### Pitfall 4: Replacing PlanSelectionCard breaks the existing polling state machine

**What goes wrong:** The existing `CheckoutReturnPoller` component is reused unchanged, but the `page.tsx` refactor accidentally removes the `session_id` state branch or changes how polling state is entered.

**Why it happens:** Refactoring `page.tsx` to replace `PlanSelectionCard` with `TierGrid` might also inadvertently touch the state derivation or polling branch.

**How to avoid:** Read the current `page.tsx` carefully. The state derivation, the `"polling"` branch, the `"active"` branch, and the `CheckoutReturnPoller` component are ALL UNCHANGED. Only the `"plan_selection"` and `"locked"` branches change (replacing `<PlanSelectionCard {...priceProps} />` with `<TierGrid ...tierProps />`).

### Pitfall 5: NSI_BRANDING_BOOKING_URL accessed client-side

**What goes wrong:** `process.env.NSI_BRANDING_BOOKING_URL` is read in a `"use client"` component. In Next.js App Router, non-`NEXT_PUBLIC_` env vars are undefined on the client bundle.

**Why it happens:** `ConsultTierCard` might be written as a client component that reads the env var directly.

**How to avoid:** Always read `process.env.NSI_BRANDING_BOOKING_URL` in a server component (`page.tsx`) and pass the value as a prop. The `ConsultTierCard` receives `bookingUrl: string` as a prop.

**Warning sign:** `bookingUrl` renders as `undefined` in the link. The hardcoded fallback in `page.tsx` is the safety net.

### Pitfall 6: Migration breaks the signup trigger

**What goes wrong:** The `provision_account_for_new_user` trigger fails after the migration because the new `plan_tier` column has a NOT NULL constraint.

**Why it happens:** Accidentally adding `NOT NULL` without a default to a column that the trigger doesn't INSERT.

**How to avoid:** The migration adds `plan_tier TEXT` (nullable, no DEFAULT). The trigger does not need to change. Postgres fills `NULL` for columns not mentioned in the INSERT.

### Pitfall 7: TypeScript narrowing fails for PriceTier type after refactor

**What goes wrong:** The checkout route previously imported `type PriceInterval` from `prices.ts`. After refactor, it also imports `type PriceTier`. If the route still uses `body.interval === 'annual' ? 'annual' : 'monthly'` coercion but tries to pass the raw `body.tier` directly to `getPriceId`, TypeScript complains because `body.tier` is `unknown`.

**How to avoid:** Explicitly narrow both:
```typescript
if (rawTier !== "basic" && rawTier !== "widget") { /* reject */ }
const tier: PriceTier = rawTier as PriceTier; // safe after the guard
const interval: PriceInterval = rawInterval === "annual" ? "annual" : "monthly";
```

## Open Questions

1. **Per-card vs global monthly/annual toggle**
   - What we know: ROADMAP says both are CONTEXT-compatible ("designer's call"). Research recommends global toggle for simplicity.
   - What's unclear: Whether Andrew has a visual preference.
   - Recommendation: Plan with global toggle (simpler); designer can override to per-card if preferred.

2. **Widget card "Most popular" badge positioning**
   - What we know: SC-2 says Widget should be marked "Most popular or equivalent." The `Badge` component supports `variant="default"` (primary color).
   - What's unclear: Whether the badge sits inside `CardTitle` inline, in `CardAction` slot, or as a standalone `CardDescription` element.
   - Recommendation: Inline in `CardHeader` next to the title using `CardAction` slot (the card component has a `data-slot="card-action"` div that positions to the right of the title naturally).

3. **Branding card feature list copy**
   - What we know: ROADMAP does not specify what features to list in the Branding card description. The Basic/Widget tiers have no feature list in the current design (per Phase 42 CONTEXT — "no feature list, user knows the product").
   - What's unclear: Whether Branding card needs any feature copy or just the heading + CTA.
   - Recommendation: Keep minimal — heading ("Branding"), subheading ("Custom design + personal onboarding"), CTA button. Planner can fill in copy if Andrew specifies.

4. **Timing of `plan_tier` write vs subscription status write**
   - What we know: `checkout.session.completed` fires first, then `customer.subscription.created`, then `invoice.payment_succeeded`. `plan_tier` is written in `checkout.session.completed`. `subscription_status='active'` is written in `invoice.payment_succeeded`.
   - What's unclear: The verification gate in SC-5 says "webhook writes the correct `plan_tier` alongside the existing `stripe_subscription_id`, `subscription_status='active'`" — this reads as if all four columns are written at the same time, but they're written in different events.
   - Recommendation: This is not an actual conflict. SC-5 is verifying that after ALL webhooks have fired, the accounts row has all four values. The "alongside" language means "in the same checkout pipeline" not "in the same DB write." The planner should write it clearly in the plan.

5. **plan_tier write ordering within handleCheckoutSessionCompleted**
   - What we know: The existing handler writes `stripe_customer_id` first. The `plan_tier` write comes after. If `listLineItems` call fails, we could throw (causing dedupe row rollback + Stripe retry) or log + continue.
   - Recommendation: Throw on `listLineItems` failure so Stripe retries the entire event. This ensures `plan_tier` is eventually written. The `stripe_customer_id` write is idempotent on retry (`.is("stripe_customer_id", null)` guard). Log a clear error distinguishing the Stripe API failure from a DB write failure.
