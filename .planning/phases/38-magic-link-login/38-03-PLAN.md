  <what-built>
The Plan 01 server action calls `supabase.auth.signInWithOtp({ email, options: { shouldCreateUser: false, emailRedirectTo: `${origin}/auth/confirm?next=/app` } })`. Without dashboard configuration, Supabase will:
- Use the default Magic Link email template, which routes to Supabase's own verify endpoint and bypasses our `/auth/confirm` PKCE handler
- Issue 1-hour-expiry OTPs, contradicting the "expires in 15 minutes" copy shown in `MagicLinkSuccess`

This task is the manual configuration these two require. It must run on the hosted Supabase project (the local `config.toml` change in Task 1 is for dev/CI parity — the dashboard is what serves preview + production).
  </what-built>
  <how-to-verify>
Andrew: complete these steps in the Supabase dashboard for the project that backs the deployed app.

**1. Set the Magic Link email template:**
   - Open Supabase Dashboard → your project → **Authentication** → **Email Templates** → **Magic Link**
   - **Subject:** `Your NSI login link`
   - **Body (replace whatever is there):**
     ```html
     <p>Click below to sign in. Link expires in 15 minutes.</p>
     <p><a href="{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=magiclink">Sign in to NSI</a></p>
     ```

   - **Critical:** the URL MUST use `{{ .TokenHash }}` and route through `/auth/confirm`. Do NOT use the default `{{ .ConfirmationURL }}` — it routes to Supabase's verify endpoint and bypasses our PKCE handler (RESEARCH §Pitfall 4).

   - Change from default (typically 3600) to **900** seconds.
   - Click Save.
   - This affects ALL email OTP flows (magic link, email-change). 15 minutes is acceptable for both per RESEARCH Open Question 1; signup confirmations are not in use (`enable_confirmations = false`).

**3. Verify the redirect URL allowlist includes /auth/confirm:**
