# Phase 42.6 Plan 01: Widget Gate Helper Summary

## Helper signature

```ts
export interface AccountGateInput {
  plan_tier: PriceTier | null;
  subscription_status: string | null;
}

export type WidgetGateResult =
  | { allowed: true }
  | { allowed: false; reason: "basic_tier" | "no_subscription" };

export function requireWidgetTier(account: AccountGateInput): WidgetGateResult;
```

Return-type shape is a discriminated union on `allowed`. Callers narrow with `if (result.allowed)` then access `result.reason` only in the false branch — TypeScript flow-narrowing enforces this at compile time.

## Branch order (CRITICAL — preserved exactly as planned)

1. `subscription_status === 'trialing'` → `{ allowed: true }` (regardless of `plan_tier`, including NULL)
2. `plan_tier === 'widget'` AND status ∈ {active, past_due} → `{ allowed: true }`
3. `plan_tier === 'basic'` AND status ∈ {active, past_due} → `{ allowed: false, reason: 'basic_tier' }`
4. Fallthrough (canceled / unpaid / incomplete / NULL-status / NULL-tier without trialing / unknown) → `{ allowed: false, reason: 'no_subscription' }`

**Trialing-first ordering is the critical invariant for Plans 02 + 03.** A new account during its 14-day trial has `plan_tier = NULL` (the webhook only writes `plan_tier` on the first paid checkout — see Phase 42.5 webhook). If the gate evaluated `plan_tier` before `subscription_status`, every trialing account would fail through to `'no_subscription'` and the public embed iframe would refuse to render for every trial user. Plans 02 and 03 MUST call this helper unmodified.
