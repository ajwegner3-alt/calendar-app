# Phase 42.6: Widget Feature Gating - Research

**Researched:** 2026-05-10
**Domain:** Next.js App Router feature gating, subscription-tier guard, embed route surface
**Confidence:** HIGH (all findings from direct codebase inspection)

## Summary

Phase 42.6 adds a lightweight gating layer over two existing surfaces: the public `/embed/[account]/[event-slug]` route and the owner-side "Get embed code" dialog entry point in `/app/event-types`. Both `plan_tier` and `subscription_status` columns already exist on the `accounts` table (written by the Phase 42.5 webhook). No new schema work is required.

The gating helper `requireWidgetTier(account)` does not exist anywhere in the codebase ŌĆö it is safe to create. The correct home for it is `lib/stripe/` (alongside the existing `prices.ts` tier logic) or a new `lib/billing/` sub-directory; either is consistent with the project's pattern of grouping domain logic under `lib/`. The embed route has no ISR/static caching ŌĆö no `export const dynamic` or `revalidate` is present ŌĆö so the route re-runs on every request, making server-side gating straightforward. The iframe framing headers are already correct: `/embed/*` strips `X-Frame-Options` and sets `frame-ancestors *` in `proxy.ts`, so gated messages will load in third-party iframes without HTTP errors.

The owner-side embed code surface lives in a **dialog** (`EmbedCodeDialog` + `EmbedTabs`), not a dedicated settings page. The "Get embed code" entry point is a `DropdownMenuItem` inside `RowActionsMenu` on the event-types list. Gating it means either: (a) making the dialog gate-aware (show upgrade card instead of tabs), or (b) disabling the menu item and showing a CTA elsewhere. Option (a) is cleaner and matches the CONTEXT decision to "replace the embed code with an upgrade CTA."

**Primary recommendation:** Create `lib/stripe/widget-gate.ts` for `requireWidgetTier()`, modify the embed page to call it server-side and render a gated message on `{ allowed: false }`, and gate the `EmbedCodeDialog` by passing a new `isWidgetAllowed` prop from the server component (`event-types/page.tsx`) down through `EventTypesTable` ŌåÆ `RowActionsMenu` ŌåÆ `EmbedCodeDialog`.

## Architecture Patterns

### Recommended File Location for Helper

```

ŌööŌöĆŌöĆ stripe/
    Ōö£ŌöĆŌöĆ prices.ts        # existing ŌĆö PriceTier type lives here
    ŌööŌöĆŌöĆ widget-gate.ts   # NEW ŌĆö requireWidgetTier()
```

Rationale: `lib/stripe/` already owns the tier model (`PriceTier`, `PRICE_ID_TO_TIER`, `priceIdToTier`). Grouping here avoids creating a new sub-directory for a single function. The helper is a pure function (no DB call) so `"server-only"` is not strictly required, but add it if the account row shape is imported from a server-only module.

### Pattern 1: Pure Gate Helper (no DB call)

The helper receives the already-fetched account row and returns a discriminated union.

```typescript
// lib/stripe/widget-gate.ts
import type { PriceTier } from "./prices";

interface AccountGateInput {
  plan_tier: PriceTier | null;
  subscription_status: string | null;
}

export type WidgetGateResult =
  | { allowed: true }
  | { allowed: false; reason: "basic_tier" | "no_subscription" };

export function requireWidgetTier(account: AccountGateInput): WidgetGateResult {
  const { plan_tier, subscription_status } = account;

  // trialing always allowed regardless of plan_tier (includes NULL)
  if (subscription_status === "trialing") return { allowed: true };

  // widget tier with active or past_due is allowed (LD-08: past_due is not lockout)
  if (
    plan_tier === "widget" &&
    (subscription_status === "active" || subscription_status === "past_due")
  ) {
    return { allowed: true };
  }

  // basic tier with active or past_due ŌåÆ gated
  if (
    plan_tier === "basic" &&
    (subscription_status === "active" || status === "past_due")
  ) {
    return { allowed: false, reason: "basic_tier" };
  }

  // everything else: canceled, unpaid, NULL plan_tier without trialing
  return { allowed: false, reason: "no_subscription" };
}
```

> Note: The planner must fix the variable name `status` to `subscription_status` in the above ŌĆö the code is illustrative.

### Pattern 2: Public Embed Route Gating (server component, always HTTP 200)

The embed page already calls `loadEventTypeForBookingPage()` which selects `id, slug, name, timezone, owner_email, logo_url, brand_primary` from `accounts`. That query must be extended to also select `plan_tier, subscription_status`. The gate check runs after the data load but before rendering `EmbedShell`.

```typescript
// app/embed/[account]/[event-slug]/page.tsx (additions)
// 1. loadEventTypeForBookingPage must SELECT plan_tier, subscription_status
// 2. After data load (before EmbedShell):
import { requireWidgetTier } from "@/lib/stripe/widget-gate";

const gate = requireWidgetTier(data.account);
if (!gate.allowed) {
  return <EmbedGatedMessage />;  // HTTP 200 ŌĆö NOT notFound()
}
// else render <EmbedShell> as today
```

**Critical invariant:** Never call `notFound()` when gated. The embed page already calls `notFound()` for missing accounts/event-types ŌĆö that is correct. Gated accounts must return a 200 with a message component.

### Pattern 3: Owner-Side Gating via Prop Threading

The embed code surface is a client component dialog launched from `RowActionsMenu`. The gating decision must be server-side. The server component `event-types/page.tsx` already fetches the account row ŌĆö it can fetch `plan_tier` and `subscription_status` there, call `requireWidgetTier`, and pass `isWidgetAllowed: boolean` as a prop down the tree.

Prop chain:
```
EventTypesPage (server) ŌĆö fetches account gate result
  ŌööŌöĆŌöĆ EventTypesTable (client, receives isWidgetAllowed)
        ŌööŌöĆŌöĆ RowActionsMenu (client, receives isWidgetAllowed)
              ŌööŌöĆŌöĆ EmbedCodeDialog (client, receives isWidgetAllowed)
```

When `!isWidgetAllowed`, `EmbedCodeDialog` shows the upgrade card instead of `EmbedTabs`.

### Anti-Patterns to Avoid

- **Calling `notFound()` when gated on `/embed/*`:** HTTP 404 breaks third-party iframes silently. Always return 200.
- **Fetching account gate data inside a client component:** The gate decision requires the `accounts` table. Client components cannot use the server Supabase client. Gate server-side, pass result as prop.
- **Caching the gated state:** The embed route has no `export const dynamic = "force-static"` ŌĆö it is already dynamic. Do not add ISR. A tier change must reflect immediately on next load.
- **Adding `plan_tier` / `subscription_status` to `AccountSummary` type indiscriminately:** These are billing fields not needed by the public booker `/{account}/{slug}` or the booking shell. Scope the extended select to the specific loaders that need it, or create a separate DB query. The bare booker MUST NOT receive these fields (no changes to its route).

## Common Pitfalls

### Pitfall 1: Extending `loadEventTypeForBookingPage` breaks the bare booker

**What goes wrong:** If `plan_tier` and `subscription_status` are added to the `select()` in `loadEventTypeForBookingPage`, the bare booker `/{account}/[event-slug]` (which shares this loader) would also receive them. If someone then naively adds gating logic to that shared function, the bare booker gets gated ŌĆö violating LD-07.

**How to avoid:** Either (a) add the gate check only in the embed page's `page.tsx` (not in the shared loader), passing a separate query for the gate columns, or (b) add `plan_tier` + `subscription_status` to the shared loader's select but gate only in the embed route's page ŌĆö never in the shared loader itself. Option (b) is simpler but requires discipline. Document the invariant explicitly.

**Warning signs:** Any gating call in `loadEventTypeForBookingPage` itself is wrong.

### Pitfall 2: Owner-side embed gating implemented as a redirect or route guard

**What goes wrong:** Adding a redirect or middleware check on `/app/event-types` would block the whole page for Basic-tier owners, preventing them from managing their event types entirely.

**How to avoid:** Gate only the embed code dialog content, not the page or the menu item. The "Get embed code" menu item can still open the dialog ŌĆö the dialog itself shows the upgrade card instead of embed tabs.

### Pitfall 3: `subscription_status` NULL treated as gated

**What goes wrong:** A newly created account (before any Stripe interaction) has `NULL` subscription_status. If the gate returns `{ allowed: false }` for NULL without checking for `trialing`, new accounts would see gated widgets immediately.

**How to avoid:** The helper's `trialing` check must run before any other check. The webhook comment in `webhook/route.ts` line 388 confirms: "plan_tier remains NULL (Phase 42.6 gating treats NULL as trialing ŌĆö allowed access)" ŌĆö honor this.

**Correct logic:** `if (subscription_status === "trialing") return { allowed: true }` runs first. NULL subscription_status (with no trialing) correctly falls through to `no_subscription`.

### Pitfall 4: Caching the embed route response

**What goes wrong:** If Next.js caches the embed route (e.g., via ISR), a tier downgrade won't be reflected until cache expiry.

**Status:** Currently safe ŌĆö no `export const dynamic` or `revalidate` exists in the embed `page.tsx`. Do not add them. The route is already fully dynamic.

### Pitfall 5: Iframe X-Frame-Options conflict

**What goes wrong:** If a gated message page sets `X-Frame-Options: SAMEORIGIN`, third-party iframes will show a broken X instead of the gated message.

**Status:** Already handled. `proxy.ts` strips `X-Frame-Options` and sets `frame-ancestors *` for all `/embed/*` routes regardless of what the page renders. The gated message component inherits this automatically.

## Testing Approach

**Existing test infrastructure:** Vitest (node + jsdom environments), no Playwright. Tests live in `tests/`. Pattern for pure helpers: unit test with `@vitest-environment node` (see `upgrade-action.test.ts` for the dependency-injection pattern used by pure helpers with mocked DB clients).

`requireWidgetTier` is a pure function (no DB, no network) ŌĆö test directly without mocks. Cover:
- `trialing` ŌåÆ `{ allowed: true }` (regardless of `plan_tier`)
- `active` + `widget` ŌåÆ `{ allowed: true }`
- `past_due` + `widget` ŌåÆ `{ allowed: true }` (LD-08)
- `active` + `basic` ŌåÆ `{ allowed: false, reason: 'basic_tier' }`
- `past_due` + `basic` ŌåÆ `{ allowed: false, reason: 'basic_tier' }`
- `canceled` (any plan_tier) ŌåÆ `{ allowed: false, reason: 'no_subscription' }`
- `null` subscription_status, `null` plan_tier ŌåÆ `{ allowed: false, reason: 'no_subscription' }`
- `trialing` + `null` plan_tier ŌåÆ `{ allowed: true }` (new account edge case)

**Local spoofing of plan_tier:** No Stripe webhook needed. Direct Supabase update in the Supabase dashboard (or a test script) against the `accounts` table is sufficient. `plan_tier` is just a string column ŌĆö set it to `'basic'` to test gating, `'widget'` for the allowed path.

## Open Questions

1. **`loadEventTypeForBookingPage` vs. separate query for gate columns**
   - What we know: the shared loader is used by both bare booker and embed route
   - What's unclear: whether adding `plan_tier, subscription_status` to the shared select would cause problems (it's additive ŌĆö bare booker would ignore the extra fields)
   - Recommendation: Add to shared loader for simplicity, but add gate call only in embed page. Add a comment to the loader stating the invariant. Simpler than a second DB query.

2. **`EmbedCodeDialog` ŌĆö hide dialog item or make dialog gate-aware?**
   - Context decision says "replace the embed code" (implying open dialog, show upgrade card)
   - Alternative: disable/hide the "Get embed code" menu item for Basic-tier owners
   - Recommendation: Make the dialog gate-aware (show upgrade card when `!isWidgetAllowed`). Consistent with CONTEXT decision wording and avoids removing functionality the owner might want to interact with.
