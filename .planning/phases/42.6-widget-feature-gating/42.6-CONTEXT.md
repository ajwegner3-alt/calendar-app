# Phase 42.6: Widget Feature Gating - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Gate the booking widget on two surfaces:

1. **Public `/embed/[account]/[slug]`** — renders a gated message (NOT 404, NOT redirect) when the account's owner is Basic-tier and not in `trialing`. iframes on third-party sites must continue to load (HTTP 200) so they don't break silently.
2. **Owner-side embed-code settings page** (existing page in `/app/*`) — replaces the embed code with an upgrade CTA for Basic-tier owners.

**Invariants:**
- `trialing` accounts always see/use the widget (no behavior change while in trial).
- `plan_tier === 'widget'` with `active`/`trialing`/(plus past_due per LD-08) status retains current behavior.
- The bare booker `/{account}/{slug}` is **never** gated (LD-07 extension; structurally exempt from paywall middleware, same exemption applies to widget gating).
- Helper `requireWidgetTier(account)` is the single source of truth — both the public embed route and the owner-side page consume it.

**Out of scope:** building the Branding tier surfaces, paywall middleware redirects (Phase 43), Customer Portal (Phase 44).

</domain>

<decisions>
## Implementation Decisions

### Claude's Discretion (the entire phase)

Andrew explicitly delegated **all four discussed areas** to Claude's judgment:

1. **Public gated message UX**
   - Copy of the gated message in the iframe
   - Whether to surface owner contact info (and which fields)
   - Iframe dimensions (match live widget vs. compact box)
   - Whether to include NSI branding/footer in the gated render

2. **Owner-side embed-code page UX**
   - Layout (full replacement vs. grayed code + banner)
   - CTA button copy ("Upgrade to Widget" vs. "Unlock booking widget" vs. "View plans")
   - CTA destination (`/app/billing` vs. anchored `/app/billing#widget`)
   - Whether to include a feature preview/blurb above the CTA

3. **Subscription state coverage**
   - How `past_due` is treated (gate vs. grace — recommend mirroring LD-08: past_due is NOT a lockout, so widget keeps working)
   - How `canceled` is treated (single message vs. distinct "paused" copy)
   - Trial-expiry behavior (immediate gate vs. grace window)
   - NULL `plan_tier` handling (recommend: trialing = allowed, everything else with NULL = `no_subscription` reason)

4. **Reason differentiation**
   - Whether end-user copy differs by reason (`basic_tier` vs. `no_subscription`)
   - Whether owner CTA copy differs by reason
   - Whether gate triggers are logged
   - Helper return shape (`{ allowed, reason }` only vs. include `displayMessage`)

### Recommended defaults Claude should adopt (subject to research findings)

These are reasonable defaults Claude should plan toward unless research reveals a reason to deviate. The planner/researcher can adjust within the discretion granted above.

**Gating logic (helper):**
- `trialing` → allowed (regardless of `plan_tier`, including NULL)
- `active` + `plan_tier === 'widget'` → allowed
- `past_due` + `plan_tier === 'widget'` → allowed (mirrors LD-08 — past_due is not a lockout)
- `active`/`past_due` + `plan_tier === 'basic'` → gated, reason `basic_tier`
- `canceled` / `unpaid` / NULL-without-trialing → gated, reason `no_subscription`

**Public gated message:**
- Same dimensions as live widget (no layout shift on embedding sites)
- Neutral, owner-agnostic copy — no NSI branding (keeps end-customer experience clean)
- No owner contact info lookup (keeps the helper/route fast and accountless-friendly)
- Single message regardless of reason (simpler, no information leakage to end-customers)

**Owner-side embed-code page:**
- Full replacement layout — hide the embed code entirely, show a single upgrade card
- CTA: "Upgrade to Widget" → links to `/app/billing` (plain, no anchor — graceful and consistent with the 3-card billing layout from 42.5)
- Short headline + 2-3 bullets + button (matches the density of surrounding settings cards)
- Same CTA regardless of reason (`basic_tier` vs. `no_subscription`) — both end at the same place

**Helper shape:**
- Return `{ allowed: boolean, reason?: 'basic_tier' | 'no_subscription' }` (matches SC-1 verbatim)
- No `displayMessage` field — keep helper purely about the gate decision; callers own their own copy
- No logging in the helper (pure function); add server-side console logging at the route level if useful for debugging

**Iframe degradation invariant:** the `/embed/[account]/[slug]` route MUST always return HTTP 200, even when gated. A 404 or redirect would silently break third-party embeds.

</decisions>

<specifics>
## Specific Ideas

- LD-07 "structural exemption" pattern from Phase 41 should extend to widget gating: the bare booker `/{account}/{slug}` is never gated, even when its owner is Basic-tier. The widget gate lives only on `/embed/*` and the owner settings page.
- The helper name `requireWidgetTier(account)` is locked by SC-1 — use that exact name.
- Reuse the existing embed-code settings page wherever it lives in `/app/*` — don't create a new route.
- Mirror LD-08 spirit (past_due = banner only, NOT lockout) for the widget gate: past_due Widget-tier accounts should keep their widget working during payment retry.

</specifics>

<deferred>
## Deferred Ideas

- Analytics/telemetry on gate triggers (which accounts hit the gate, how often) — separate observability phase if it becomes useful.
- A/B testing different upgrade CTA copy — separate growth phase.
- Custom per-account "paused" copy that owners can author themselves — out of scope for v1.8.
- Email notification to owner when their widget gates for the first time after Basic-tier subscription — could be added in Phase 44 (Stripe-triggered emails) if it fits naturally.
- Branding-tier-specific surfaces (the consult CTA is locked for billing only; no Branding-tier widget logic exists since Branding never writes `plan_tier`).

</deferred>
