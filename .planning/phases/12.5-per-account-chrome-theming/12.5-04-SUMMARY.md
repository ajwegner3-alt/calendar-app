## Decisions Made

- **Intensity='none' email behavior (EMAIL-13 interpretation):** When an owner sets intensity='none', their UI chrome is gray-50 (no color tint). To match, the email header reverts to `brand_primary` (not `background_color`). This makes "no tint" mean "no color in header band" on both surfaces.
- **'subtle' and 'full' both use full backgroundColor in email:** Email clients cannot render `color-mix()`. The distinction between subtle/full only matters in the UI (CSS color-mix percentages). In email, both intensity levels that "use color" use the direct `background_color` hex value.
- **Optional field + 'subtle' default for backward compat:** Making `chromeTintIntensity` optional in `EmailBranding` means existing callers (tests, any future sender that hasn't been updated) behave exactly as in Phase 12 — `background_color` in the header band, same as before.
- **Type cast for DB string → union:** The DB returns `string | null`; `EmailBranding` wants `"none"|"subtle"|"full"|undefined`. Using `(account.chrome_tint_intensity as "none"|"subtle"|"full"|undefined) ?? "subtle"` handles null values and avoids a TS error without introducing runtime risk (the DB column is constrained by a check constraint from 12.5-01).
- **BRAND-09 verified:** `pickTextColor` is already in `lib/branding/contrast.ts` and already imported by `branding-blocks.ts` since Plan 12-01. No extraction needed; documented here as a no-op confirmation.
