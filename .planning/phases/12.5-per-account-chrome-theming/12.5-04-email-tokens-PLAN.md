<objective>
Extend the email senders so the header band derives from chrome_tint_intensity, completing
the "same tokens for UI and email" requirement (EMAIL-13).

Purpose: After Phase 12.5 ships, an owner who sets chrome_tint_intensity='none' should see
their email header revert to using brand_primary (not background_color) — matching the
"gray-50 chrome = neutral branding" contract. An owner who sets intensity='subtle' or 'full'
keeps the existing Phase 12 behavior (background_color in the header band). This ties
email rendering to the exact same intensity semantics as the dashboard chrome.

Architectural decision committed (EMAIL-13 interpretation):
- intensity='none': header band uses brand_primary (or DEFAULT_BRAND_PRIMARY). The owner
  chose "no color tint in the UI" — the email should match: no background_color in the band.
- intensity='subtle' or 'full': header band uses backgroundColor (current Phase 12 behavior).
  Both intensity levels use the full backgroundColor in email (not a partial tint like the UI
  uses color-mix). Email clients cannot render color-mix(); we use the direct hex value.
  The intensity distinction in email is only about whether background_color is used or not.
- This preserves full backward compatibility: existing accounts with default 'subtle' continue
  to use backgroundColor in the header band (same as Phase 12 behavior).

No new user-facing fields. No email-specific branding controls. EMAIL-13 locked.

Output: branding-blocks.ts EmailBranding extended + renderEmailBrandedHeader intensity-aware;
all 6 senders pass chromeTintIntensity from account row.
</objective>
