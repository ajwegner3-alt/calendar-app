## Architectural Decisions for the Planner

### Decision 1: Tinting strategy — color-mix vs. opacity overlay vs. dual-token

The chrome tint needs to be derived from `account.background_color` (or fallback `brand_primary`) but apply at very low intensity to surfaces. Three strategies:

- **(a) `color-mix(in oklch, ${color} N%, white)`** — same primitive `shadeToGradient` already uses for `shade='none'` flat tint. Composable, no opacity issues. Recommended.
- **(b) Color + low alpha overlay div** — easy but compounds with content opacity unpredictably.
- **(c) Pre-computed token table** (e.g., 8 swatches × 3 intensities = 24 tints baked at build) — defeats the purpose of allowing custom hex.

Planner should pick (a) and codify the percentage-per-intensity-per-surface in a single helper. Suggested:
- Sidebar bg: `intensity=full → color-mix 14%`, `subtle → 6%`, `none → gray-50`
- Page bg:    `intensity=full → color-mix 8%`, `subtle → 3%`, `none → gray-50`
- Card bg:    KEEP white at all intensities (preserves Cruip card-on-tint readability)

### Decision 2: WCAG helper location

Existing email pickTextColor lives in `lib/email/branding-blocks.ts` (or similar). Extract to `lib/branding/contrast.ts` (already exists per Phase 12 imports — verify) and consume from both UI and email paths.

### Decision 3: Header pill removal — relocate trigger or drop entirely?

Mobile needs SOME way to open the sidebar drawer. Options:
- (a) Move SidebarTrigger to top-left of the sidebar drawer header itself (sidebar opens itself when tapped)
- (b) Add a small in-page anchor at top of every (shell) route's main element
- (c) Keep a minimal hamburger button in `(shell)/layout.tsx` that's NOT a glass pill — just a plain button
Planner picks (c) most likely (least surprising UX), but justify the choice.

### Decision 4: Branding editor preview semantics

`mini-preview-card.tsx` currently shows the gradient backdrop only. Phase 12.5 needs it to also show a faux-sidebar + faux-page-bg + faux-card so the owner can see chrome theming live. Planner decides whether to:
- (a) Replace the gradient-only preview with a chrome-aware preview
- (b) Add a SECOND preview card next to the gradient one
- Recommended: (a) — single source of truth, less visual noise.
