<objective>
Apply per-account chrome tinting to the dashboard sidebar and page background, and remove
FloatingHeaderPill from every dashboard route.

Purpose: This is the core visual change of Phase 12.5. An owner who sets brand_primary +
background_color + chrome_tint_intensity='full' will see their brand color wash the entire
dashboard chrome (sidebar bg, page bg) with auto-WCAG text contrast. An owner who keeps
the default 'subtle' gets a light accent. 'none' is a regression-safe path back to Phase 12
gray-50 chrome for owners who prefer neutral.

Architectural decisions committed:
- Sidebar tinting: inline `style={{ backgroundColor: chromeTintToCss(...) ?? undefined }}` on
  the `<Sidebar>` root element in app-sidebar.tsx. When chromeTintToCss returns null (none
  intensity or no color), the `?? undefined` removes the style and the component falls back to
  its existing CSS class default (sidebar background color token). No dynamic Tailwind classes.
- Page bg tinting: same pattern on SidebarInset in shell layout.
- WCAG text flip: sidebar nav items receive inline color from chromeTintTextColor. When the
  function returns null (no tint), existing text color classes apply. When it returns '#000000'
  or '#ffffff', it's applied via inline style only.
- Header trigger relocation: Decision (c) from CONTEXT.md. FloatingHeaderPill deleted.
  A `<SidebarTrigger>` is placed in a small fixed-position area inside the SidebarInset
  (top-left, outside the main content max-width, z-20). This avoids the glass pill competing
  with the tinted chrome while keeping mobile trigger accessible.
- Card bg: always white. Do NOT tint cards. Only sidebar and SidebarInset page bg.

Output: Updated layout.tsx and app-sidebar.tsx with inline tinting; floating-header-pill.tsx
deleted; plain hamburger trigger in layout.
</objective>
