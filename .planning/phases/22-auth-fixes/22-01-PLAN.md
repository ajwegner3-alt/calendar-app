<objective>
Fix two surgical UI/middleware bugs on the owner-auth surfaces:

1. **AUTH-18:** The "Sign up" link on `/app/login` is broken — clicking it bounces the user back to `/app/login`. The href is correct (`/app/signup`) and uses Next.js `<Link>` properly. The bug is in `lib/supabase/proxy.ts`: the unauthenticated-redirect rule only exempts `/app/login`, so unauthenticated visits to `/app/signup` (and `/app/forgot-password`, `/app/verify-email`) are intercepted and redirected back to login. Fix by widening the allow-list.

2. **AUTH-19:** The desktop `/app/login` layout currently renders the form on the LEFT and `AuthHero` on the RIGHT. ROADMAP says the info pane should be on the LEFT and the form on the RIGHT. Swap the JSX children inside the existing `lg:grid-cols-2` div. No class changes needed; `AuthHero` is `hidden lg:flex` so mobile is unaffected.

Per CONTEXT.md ("anchored to existing v1.2 auth skin") and RESEARCH.md ("preserve the existing `lg:` breakpoint"), this plan keeps `lg:`. This is a doc-vs-code discrepancy, not a regression — flag for verifier awareness.

Note: The reciprocal "Already have an account? Log in" link on `/app/signup` already exists in `signup-form.tsx` (research confirmed). No work needed.

Purpose: Owner can reach the signup flow from login (currently impossible to onboard a new user without typing the URL directly), and the login page matches the intended Cruip-style split-panel direction.

Output: Two file edits — middleware allow-list expansion + JSX child swap. Total ~10 lines of changes.

  <name>Task 1: Widen middleware unauthenticated-route allow-list (AUTH-18)</name>
  <files>lib/supabase/proxy.ts</files>
  <action>
Edit `lib/supabase/proxy.ts` to replace the single-route exemption with an array allow-list.

Current code (lines 47–56):
```ts
const { pathname } = request.nextUrl;
if (
  !user &&
  pathname.startsWith("/app") &&
  pathname !== "/app/login"
) {
  const url = request.nextUrl.clone();
  url.pathname = "/app/login";
  return NextResponse.redirect(url);
}
```

Replace with:
```ts
const { pathname } = request.nextUrl;
// Public auth routes — must be reachable without a session so unauthenticated
// users can sign up, recover passwords, and confirm email tokens.
// AUTH-18 (v1.3): /app/signup was broken because only /app/login was exempted.
const publicAuthPaths = [
  "/app/login",
  "/app/signup",
  "/app/forgot-password",
  "/app/verify-email",
];
if (
  !user &&
  pathname.startsWith("/app") &&
  !publicAuthPaths.includes(pathname)
) {
  const url = request.nextUrl.clone();
  url.pathname = "/app/login";
  return NextResponse.redirect(url);
}
```

Why these four paths:
- `/app/login` — existing exemption; preserved.
- `/app/signup` — the AUTH-18 fix; new-user signup must be reachable without auth.
- `/app/forgot-password` — same class of public auth route; if a user forgot their password they have no session.
- `/app/verify-email` — email-confirmation deep links land here; user may not yet have a session.

Use `=== / .includes()` (exact match) NOT `.startsWith()` — we don't want to accidentally exempt a future `/app/signupthing` route. All four routes are exact pathnames per the App Router file structure (`app/(auth)/app/{login,signup,forgot-password,verify-email}/page.tsx`).

Do NOT introduce additional behavioral changes (don't touch the `setAll` cookies callback, don't touch the env-var bootstrap guard, don't touch the `getClaims()` call). Plan 22-02 owns the `setAll` headers correctness fix — keep this change scoped to the gate condition.
