<objective>
Wire every email-sending path in the system through the quota guard. Refuse-send fail-closed at 200/day for booking confirmations, owner notifications, reminders (cron + manual + immediate), cancels, and reschedules. Save-and-flag for booker confirmations (no rollback of saved booking; flag the row for the dashboard alert in Plan 31-03). Switch cancel/reschedule from `after()` to `await` for the quota check so errors bubble synchronously to the owner UI.

Purpose: This plan closes the v1.1 carve-out. Currently the booking-side senders bypass the guard entirely — the codebase comment in `lib/email-sender/index.ts` states "Booking and reminder paths bypass the guard intentionally." This is the line that gets deleted. After this plan, every send goes through `checkAndConsumeQuota` with a typed EmailCategory and refuses at the cap with a structured PII-free log line.

Output: 7 email-sending functions guarded, 2 lifecycle modules switched from after() to await, 1 cron route handling mid-batch refusal cleanly, 1 server action returning quota-specific error copy, the carve-out comment removed.
</objective>

**A. `lib/email/send-booking-emails.ts` — save-and-flag for booker confirmation refusal:**

Currently the function does roughly:
```typescript
const [confResult, ownerResult] = await Promise.allSettled([
  sendBookingConfirmation(...),
  sendOwnerNotification(...),
]);
// log per-leg failures, swallow
```

Phase 31 change: After the allSettled, inspect each rejected result. If EITHER leg's reason is a `QuotaExceededError`, do a service-role UPDATE on the booking row setting `confirmation_email_sent = false`. Use the existing admin client pattern (`createAdminClient()` from `@/lib/supabase/admin` — same client the rest of the email layer uses for service-role writes). The UPDATE is fire-and-forget within `after()` — it's already a post-response code path, so a small extra DB write is fine.

```typescript
import { QuotaExceededError } from "@/lib/email-sender/quota-guard";
import { createAdminClient } from "@/lib/supabase/admin";

const [confResult, ownerResult] = await Promise.allSettled([
  sendBookingConfirmation(...),
  sendOwnerNotification(...),
]);

const quotaHit =
  (confResult.status === "rejected" && confResult.reason instanceof QuotaExceededError) ||
  (ownerResult.status === "rejected" && ownerResult.reason instanceof QuotaExceededError);

if (quotaHit) {
  const admin = createAdminClient();
  const { error: flagErr } = await admin
    .from("bookings")
    .update({ confirmation_email_sent: false })
    .eq("id", bookingId);
  if (flagErr) {
    console.error("[EMAIL_QUOTA_EXCEEDED] failed to flag booking", { booking_id: bookingId, error: flagErr.message });
  }
}

// keep existing per-leg .catch logging for non-quota failures
```

The held slot is NOT released. The booking row stays. Per RESEARCH "Held-slot fate: no change."

**B. `lib/bookings/cancel.ts` — switch from after() to await:**

Currently:
```typescript
after(() => sendCancelEmails(...));
return { ok: true };
```

Change to:
```typescript
try {
  await sendCancelEmails(...);
  return { ok: true };
} catch (err) {
  if (err instanceof QuotaExceededError) {
    return { ok: true, emailFailed: "quota" as const };
  }
  // Non-quota errors: log + still return ok (booking is cancelled in DB regardless of email outcome)
  console.error("[CANCEL_EMAIL_FAILED]", { booking_id: bookingId, error: err instanceof Error ? err.message : String(err) });
  return { ok: true, emailFailed: "send" as const };
}
```

Update the return type signature so callers (the cancel server action / cancel-button.tsx) can branch on `emailFailed`. Note: `Promise.allSettled` inside `sendCancelEmails` does NOT re-throw — so we must explicitly inspect the settled results inside `sendCancelEmails` and throw the FIRST QuotaExceededError encountered. Add at the end of `sendCancelEmails`:

```typescript
const quotaErr = results.find(
  (r): r is PromiseRejectedResult => r.status === "rejected" && r.reason instanceof QuotaExceededError,
);
if (quotaErr) throw quotaErr.reason;
```

**D. `app/api/cron/send-reminders/route.ts` — per-booking quota check + continue + 200 exit:**

Inside the `after(async () => { for (const c of claimed) { ... } })` block, the existing per-booking try/catch already exists. Extend it:

```typescript
let remindersSent = 0;
let quotaRefused = 0;

for (const c of claimed) {
  try {
    await sendReminderBooker({ ... }); // already calls checkAndConsumeQuota('reminder') internally
    remindersSent++;
  } catch (err) {
    if (err instanceof QuotaExceededError) {
      quotaRefused++;
      // logQuotaRefusal already called inside sendReminderBooker — do NOT double-log
      // Do NOT clear reminder_sent_at — v1 posture per RESEARCH "Cron Behavior Design"
      continue; // keep iterating; remaining bookings will also refuse and accumulate the count
    }
    console.error("[REMINDER_SEND_FAILED]", { booking_id: c.id, error: err instanceof Error ? err.message : String(err) });
  }
}

// (Outside after(), in the response):
return Response.json({ ok: true, scanned, claimed: claimed.length, reminders_sent: remindersSent, quota_refused: quotaRefused });
```

Note: the response is returned BEFORE `after()` runs (`remindersSent` and `quotaRefused` will be 0 at response time because the loop hasn't started). The actual numbers can be observed only via Vercel Functions logs. Two acceptable approaches:
- (a) Move the loop OUT of `after()` so the counters are live at response time.
- (b) Keep `after()` and emit the counters only via console.log at end of loop.

**Pick (a)** for Phase 31: the cron is server-internal (cron-job.org or Vercel Cron), there is no human waiting on the response, and live counters in the JSON response give better operability. Remove the `after()` wrapper around the loop. The response will return after the loop completes.
