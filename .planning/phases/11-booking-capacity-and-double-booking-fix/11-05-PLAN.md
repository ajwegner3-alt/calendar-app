**Change 2: Fix Pitfall 4 — change the bookings status filter.**

```typescript
// BEFORE (current — over-blocks; rescheduled bookings prevent re-use of the freed slot):
.neq("status", "cancelled")
// AFTER (correct — only confirmed bookings consume capacity / exclude slots):
.eq("status", "confirmed")
```

**Why this is correct:** A 'rescheduled' booking represents an OLD slot that was freed when the booker rescheduled. Per RESEARCH.md §Pitfall 4, the v1.0 `.neq('cancelled')` filter was over-blocking because it lumped 'rescheduled' with 'confirmed'. After this fix, a 'rescheduled' row at slot S no longer prevents a fresh booking at S — which is the intended cancel/reschedule contract.

**Side-effect verification:** This change interacts with the existing `bookings_no_double_book` (now replaced by `bookings_capacity_slot_idx`) — the index condition is `WHERE status='confirmed'`, so the slot uniqueness guard ALSO uses 'confirmed' only. The semantic alignment is intentional and locked.

**Change 3: Pass capacity fields into computeSlots.** Find the `computeSlots(...)` call. Add the two new fields to the input object:

```typescript

```
