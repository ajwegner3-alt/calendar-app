**Change 2: Add the CAP-09 over-cap pre-check.** This runs BEFORE the upsert, ONLY when:
1. This is an UPDATE (not a create — new event types have no bookings yet); detect via the presence of an existing event_type id.
2. The new `max_bookings_per_slot` is LESS than the row's current value (capacity decrease).
3. `confirmCapacityDecrease` is FALSE (owner has not yet acknowledged the warning).

Pseudocode:
```typescript
// After Zod parse, before upsert. Detect "this is an update of an existing event type":
if (existingEventTypeId && parsed.confirmCapacityDecrease !== true) {
  // Fetch current capacity to compare:
  const { data: currentRow } = await supabase
    .from("event_types")
    .select("max_bookings_per_slot")
    .eq("id", existingEventTypeId)
    .single();

  const currentCap = currentRow?.max_bookings_per_slot ?? 1;
  const newCap = parsed.max_bookings_per_slot;

  if (newCap < currentCap) {
    // Run the over-cap aggregate per RESEARCH.md §Pitfall 6:
    // count confirmed FUTURE bookings per slot for this event type;
    // if any slot has more than newCap, return a warning instead of saving.
    const { data: overCapRows, error: overCapErr } = await supabase
      .from("bookings")
      .select("start_at", { count: "exact", head: false })
      .eq("event_type_id", existingEventTypeId)
      .eq("status", "confirmed")
      .gt("start_at", new Date().toISOString());
    // NOTE: supabase-js doesn't support GROUP BY/HAVING directly. Use a Postgres RPC
    // or count rows in JS as a workaround. Since the volume of future bookings per
    // event type is small (one owner's data, prefetched and grouped client-side is fine),
    // group by start_at in JS and check if any group has count > newCap.

    if (overCapErr) {
      // Defensive: fail closed — surface a generic error rather than silently saving.
      return { ok: false, error: "Could not verify capacity change. Please try again." };
    }

    const slotCounts = new Map<string, number>();
    for (const row of overCapRows ?? []) {
      slotCounts.set(row.start_at, (slotCounts.get(row.start_at) ?? 0) + 1);
    }

    let maxAffected = 0;
    let affectedSlots = 0;
    for (const cnt of slotCounts.values()) {
      if (cnt > newCap) {
        affectedSlots++;
        if (cnt > maxAffected) maxAffected = cnt;
      }
    }

    if (affectedSlots > 0) {
      // Return a structured warning, not an error. The form component shows the modal.
      return {
        ok: false,
        warning: "capacity_decrease_overflow",
        details: {
          newCap,
          currentCap,
          affectedSlots,    // how many distinct future slots exceed the new cap
          maxAffected,      // the worst-case slot's confirmed count
        },
      };
    }
  }
}
```

**Adapt to the file's actual return shape.** If the existing actions return `{ ok: true, data }` or `{ ok: false, error }`, add a third variant `{ ok: false, warning: "...", details: {...} }`. If the existing pattern uses thrown errors, throw a typed error class instead.

**Change 3: Use `.lt("start_at", ...)` if your "future" definition differs.** RESEARCH.md uses `start_at > now()`. If the project elsewhere defines "future" via a different threshold (e.g., end-of-day), mirror that. Default to `> now()`.

**Why prefetch + group in JS instead of an RPC:** RESEARCH.md §Pitfall 6 shows the SQL pattern, but supabase-js doesn't expose GROUP BY/HAVING natively. The data volume is tiny (one owner's future bookings for one event type — typically <100 rows). Doing the group-by in JS is acceptable and avoids creating a new RPC.

**If you prefer an RPC** (future-proofing for owners with large booking volumes), create a `count_overcap_future_bookings(event_type_id, new_cap)` SECURITY INVOKER function. This is an acceptable alternative — document the choice in the SUMMARY.
