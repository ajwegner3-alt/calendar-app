# Phase 12.6: Direct Per-Account Color Controls (Page / Sidebar / Primary)

**Inserted:** 2026-04-29
**Origin:** Andrew reviewed the Phase 12.5 deploy on Vercel and reported three colors that should be per-account-changeable but aren't — or are technically settable but don't render in a way that looks like the picked color:
1. The page background looks chrome (gray) regardless of his picks
2. The sidebar is white regardless of his picks
3. The "tertiary" color (button bg on event types, switch bg on availability) is black — driven by shadcn's default `--primary`, NOT by `account.brand_primary`

**Why Phase 12.5 didn't deliver this:**
Phase 12.5 used `color-mix(in oklch, ${color} N%, white)` with low percentages (sidebar full=14%, page full=8%). At those percentages, even a strong navy looks "barely tinted off-white" — visually indistinguishable from the gray-50 default. The intensity picker conflated "subtle accent" with "actually adopt my color." Andrew wants the latter: when he picks navy, the sidebar should be navy, not 14%-navy-86%-white.

Additionally, `account.brand_primary` (which has existed since v1.0) has never been wired to override shadcn's `--primary` CSS variable in the (shell) scope. It only feeds the public booking page's slot accents, calendar dots, and email header band — not the dashboard's button/switch/focus-ring color.

## Phase 12.6 Goal

A dashboard owner can independently configure THREE per-account colors and see each one render at full strength on the surface it controls, with auto-WCAG text contrast where applicable:
- **Sidebar color** — the actual fill color of the dashboard sidebar
- **Page background color** — the actual fill color of the dashboard page area
- **Primary color** (shadcn `--primary` override) — drives button bg, switch on-state, focus rings, and any other shadcn-primary consumer

## Andrew's Locked Decisions (2026-04-29)

1. **Three distinct color fields**, all per-account, all settable to any hex (or null = default).
2. **Direct rendering, not percentage tinting.** When Andrew picks `#0A2540` for sidebar, the sidebar bg is `#0A2540`, not `color-mix(navy 14%, white)`.
3. **Auto-WCAG text contrast** on each tinted surface (already exists; `pickTextColor` in `lib/branding/contrast.ts`).
4. **Per-user (per-account)**, not a global switch.

## CSS Primary Override Strategy

The shadcn `--primary` CSS variable is set in `app/globals.css` `:root` selector (typically `oklch(...)` form per shadcn v4 conventions). To override per-account inside the (shell) scope only:

```tsx
// app/(shell)/layout.tsx
<div
  style={{
    backgroundColor: pageColor ?? undefined,
    "--primary": brandPrimary ?? undefined,
    "--primary-foreground": brandPrimary ? pickTextColor(brandPrimary) : undefined,
  } as React.CSSProperties}
>
  {/* sidebar + content */}
</div>
```

CSS-variable inheritance applies the new `--primary` to all shadcn components within the shell. The public `/[account]` and `/embed` surfaces are NOT in the shell scope and continue using their own brand color application paths (Phase 7 BrandedPage CSS vars).

**Phase 7 pitfall LOCKED** — never Tailwind dynamic classes for runtime hex; inline `style` and CSS-variable inline-style only.

**globals.css preserves the default** — do NOT change globals.css `--primary`. The override is purely scoped via inline style.
